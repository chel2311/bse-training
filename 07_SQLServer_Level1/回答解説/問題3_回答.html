バグ1: 在庫チェックがない（BR-003違反）

該当仕様: BR-003 在庫チェック

受注登録時に在庫数が注文数量を下回る場合はエラーとする。
条件: Products.StockQuantity >= OrderDetails.Quantity
違反時: エラーを返し、受注登録を行わない

また、ストアドプロシージャ仕様「6.1 sp_CreateOrder」処理フロー:

  1. 在庫チェック（BR-003）: StockQuantity >= @Quantity を確認

該当コード: sp_CreateOrder.sql 在庫減算処理の前

-- 現在の実装（バグ）

-- 商品の単価を取得
SELECT @UnitPrice = UnitPrice
FROM Products
WHERE ProductID = @ProductID;

-- 商品が存在しない場合のチェックのみ
IF @UnitPrice IS NULL
BEGIN
    RAISERROR(...);
    RETURN;
END

-- ← ここに在庫チェックがない。すぐに小計計算・INSERT処理に進んでいる

SET @Subtotal = @Quantity * @UnitPrice;

-- ... INSERT処理 ...

-- 在庫を減算（在庫数が足りなくてもそのまま実行される）
UPDATE Products
SET StockQuantity = StockQuantity - @Quantity
WHERE ProductID = @ProductID;

修正コード（商品存在チェックの後に追加）:

-- 商品が存在しない場合
IF @UnitPrice IS NULL
BEGIN
    RAISERROR(N'指定された商品が存在しません。ProductID: %d', 16, 1, @ProductID);
    RETURN;
END

-- 在庫チェック（BR-003）
DECLARE @CurrentStock INT;
SELECT @CurrentStock = StockQuantity
FROM Products
WHERE ProductID = @ProductID;

IF @CurrentStock < @Quantity
BEGIN
    RAISERROR(N'在庫が不足しています。商品ID: %d, 在庫数: %d, 注文数量: %d',
              16, 1, @ProductID, @CurrentStock, @Quantity);
    RETURN;
END

レビュー指摘例:

【指摘箇所】sp_CreateOrder.sql: 商品存在チェック後～INSERT処理前
【分類】仕様不一致
【重要度】高
【内容】設計仕様書 BR-003 では「在庫数が注文数量を下回る場合はエラー」と
        定義されており、ストアドプロシージャ仕様の処理フロー第1ステップでも
        「在庫チェック」が明記されているが、実装では在庫数の確認処理がない。
        在庫数が不足していてもそのまま減算が実行され、
        マイナス在庫が発生する。CHECK制約がある場合はSQLエラーになるが、
        適切なエラーメッセージが返されない。
【修正案】商品存在チェックの後に、StockQuantity >= @Quantity の判定を追加し、
        不足時はRAISERRORで在庫不足エラーメッセージを返してRETURNしてください。

バグ2: トランザクション制御がない

該当仕様: ストアドプロシージャ仕様「6.1 sp_CreateOrder」

上記の処理はすべてトランザクション内で実行すること

エラー処理:

その他のエラー: ロールバックしてエラーメッセージを返す

該当コード: sp_CreateOrder.sql 全体

-- 現在の実装（バグ）

-- ← BEGIN TRANSACTION がない

-- Ordersテーブルに受注ヘッダを挿入
INSERT INTO Orders (...) VALUES (...);

SET @NewOrderID = SCOPE_IDENTITY();

-- OrderDetailsテーブルに受注明細を挿入
INSERT INTO OrderDetails (...) VALUES (...);

-- 在庫を減算
UPDATE Products SET StockQuantity = StockQuantity - @Quantity ...;

-- ← COMMIT TRANSACTION がない
-- ← TRY-CATCH による ROLLBACK がない

問題点:
- Orders INSERT成功後、OrderDetails INSERTでエラーが発生した場合、
Ordersにだけレコードが残り、明細のない不整合な受注データになる
- OrderDetails INSERT成功後、在庫UPDATE でエラーが発生した場合、
受注は登録されたが在庫が減っていない状態になる

修正コード:

BEGIN TRY
    BEGIN TRANSACTION;

    -- Ordersテーブルに受注ヘッダを挿入
    INSERT INTO Orders (CustomerID, OrderDate, TotalAmount, Status)
    VALUES (@CustomerID, GETDATE(), 0, N'未処理');

    SET @NewOrderID = SCOPE_IDENTITY();

    -- OrderDetailsテーブルに受注明細を挿入
    INSERT INTO OrderDetails (OrderID, ProductID, Quantity, UnitPrice, Subtotal)
    VALUES (@NewOrderID, @ProductID, @Quantity, @UnitPrice, @Subtotal);

    -- 在庫を減算（BR-002）
    UPDATE Products
    SET StockQuantity = StockQuantity - @Quantity
    WHERE ProductID = @ProductID;

    -- TotalAmountを更新（BR-001）
    UPDATE Orders
    SET TotalAmount = (
        SELECT SUM(Subtotal) FROM OrderDetails WHERE OrderID = @NewOrderID
    )
    WHERE OrderID = @NewOrderID;

    COMMIT TRANSACTION;
END TRY
BEGIN CATCH
    IF @@TRANCOUNT > 0
        ROLLBACK TRANSACTION;

    DECLARE @ErrorMessage NVARCHAR(4000) = ERROR_MESSAGE();
    RAISERROR(@ErrorMessage, 16, 1);
END CATCH
トランザクションなしで途中エラーが発生した場合
Orders INSERT → 成功
↓
OrderDetails INSERT → エラー発生
↓
Ordersにだけレコードが残る（不整合）
[OK] トランザクションありの場合
BEGIN TRANSACTION
↓
Orders INSERT → 成功
↓
OrderDetails INSERT → エラー発生
↓
ROLLBACK → 全て取り消し（整合性維持）

レビュー指摘例:

【指摘箇所】sp_CreateOrder.sql: ストアドプロシージャ全体
【分類】仕様不一致
【重要度】高
【内容】設計仕様書では「すべてトランザクション内で実行すること」と
        定義されているが、実装ではBEGIN TRANSACTION / COMMIT / ROLLBACK が
        一切記述されていない。
        Orders INSERT → OrderDetails INSERT → 在庫UPDATE の途中で
        エラーが発生した場合、処理が中途半端な状態で停止し、
        データの整合性が破壊される。
        例: OrdersにレコードがあるがOrderDetailsが空、
            受注は登録されたが在庫が減っていない、など。
【修正案】INSERT/UPDATE処理全体をBEGIN TRY / BEGIN TRANSACTIONで囲み、
        正常時はCOMMIT、エラー時はCATCHブロックでROLLBACKしてください。

バグ3: TotalAmount の更新漏れ（BR-001違反）

該当仕様: BR-001 受注合計金額の計算

Orders.TotalAmount = 該当OrderIDのOrderDetails.Subtotalの合計（SUM）
受注明細が追加・変更・削除されるたびに、OrdersのTotalAmountを再計算して更新すること。

また、ストアドプロシージャ仕様「6.1 sp_CreateOrder」処理フロー:

  1. TotalAmountを更新（BR-001）

該当コード: sp_CreateOrder.sql 在庫減算処理の後

-- 現在の実装（バグ）

-- 在庫を減算（BR-002）
UPDATE Products
SET StockQuantity = StockQuantity - @Quantity
WHERE ProductID = @ProductID;

-- ← ここでOrdersのTotalAmountを更新する処理がない
-- TotalAmountは INSERT時に 0 で登録されたまま

PRINT N'受注を登録しました。OrderID: ' + CAST(@NewOrderID AS NVARCHAR(10));

修正コード（在庫減算の後に追加）:

-- 在庫を減算（BR-002）
UPDATE Products
SET StockQuantity = StockQuantity - @Quantity
WHERE ProductID = @ProductID;

-- TotalAmountを更新（BR-001）
UPDATE Orders
SET TotalAmount = (
    SELECT ISNULL(SUM(Subtotal), 0)
    FROM OrderDetails
    WHERE OrderID = @NewOrderID
)
WHERE OrderID = @NewOrderID;

レビュー指摘例:

【指摘箇所】sp_CreateOrder.sql: 在庫減算処理後（ストアドプロシージャ末尾）
【分類】仕様不一致
【重要度】高
【内容】設計仕様書 BR-001 では「TotalAmount = OrderDetailsのSubtotalの合計」と
        定義されており、ストアドプロシージャ仕様の処理フロー第5ステップでも
        「TotalAmountを更新」と明記されているが、
        実装ではOrderDetails挿入後にOrdersのTotalAmountを更新する処理がない。
        TotalAmountはINSERT時に0で登録されたまま残り、
        帳票やクエリでの金額集計が正しく行われない。
【修正案】在庫減算処理の後に、OrderDetailsのSubtotalをSUMして
        OrdersのTotalAmountを更新するUPDATE文を追加してください。

バグの見つけ方のコツ

  1. ビジネスルールを1つずつ照合する
    - 仕様書のBR-001〜BR-005を順番にチェック
    - 各ビジネスルールに対応する実装があるか確認
    - 今回のバグ1（BR-003）とバグ3（BR-001）はこの照合で発見できる

  2. 処理フローを仕様書と突き合わせる
    - 仕様書の「処理フロー」を上から順に実装と比較
    - 抜けている手順がないか確認
    - 手順の実行順序が正しいか確認

  3. トランザクション制御は最重要チェック項目
    - 複数テーブルを更新するストアドプロシージャには必ずトランザクションが必要
    - 「途中で失敗したらどうなるか」をシミュレーションする
    - バグ2はこの観点で発見できる

  4. 「初期値のまま残る」パターンに注意
    - INSERT時にデフォルト値を入れた後、正しい値に更新する処理が漏れていないか
    - 今回のTotalAmount = 0 のまま残るバグ3がこのパターン
