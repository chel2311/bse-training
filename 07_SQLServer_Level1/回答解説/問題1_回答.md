### バグ1: OrdersテーブルのCustomerIDに外部キー制約がない

**該当仕様**: BR-005 外部キー制約の必須設定
> **CustomerID、ProductID、OrderIDの参照関係には、必ず外部キー制約（FOREIGN KEY）を設定すること。**

また、テーブル設計「3.3 Orders」:
> CustomerID: **FOREIGN KEY → Customers(CustomerID)**

**該当コード**: `CreateTables.sql` Ordersテーブル定義

```sql
-- 現在の実装（バグ）
CREATE TABLE Orders (
    OrderID      INT            PRIMARY KEY IDENTITY(1,1),
    CustomerID   INT            NOT NULL,
    OrderDate    DATE           NOT NULL DEFAULT GETDATE(),
    TotalAmount  DECIMAL(12,2)  NOT NULL DEFAULT 0,
    Status       NVARCHAR(20)   NOT NULL DEFAULT N'未処理'
    -- ← CustomerIDへの外部キー制約がない
);
```

**修正コード**:
```sql
CREATE TABLE Orders (
    OrderID      INT            PRIMARY KEY IDENTITY(1,1),
    CustomerID   INT            NOT NULL,
    OrderDate    DATE           NOT NULL DEFAULT GETDATE(),
    TotalAmount  DECIMAL(12,2)  NOT NULL DEFAULT 0,
    Status       NVARCHAR(20)   NOT NULL DEFAULT N'未処理',
    CONSTRAINT FK_Orders_Customers
        FOREIGN KEY (CustomerID) REFERENCES Customers(CustomerID)
);
```

**レビュー指摘例**:
```
【指摘箇所】CreateTables.sql: OrdersテーブルのCREATE TABLE定義
【分類】仕様不一致
【重要度】高
【内容】設計仕様書 BR-005 および テーブル設計3.3 では、
        Orders.CustomerID に Customers(CustomerID) への外部キー制約を
        設定すると定義されているが、実装では外部キー制約が定義されていない。
        存在しないCustomerIDの受注レコードが登録可能な状態であり、
        データの整合性が保証されない。
【修正案】OrdersテーブルにCONSTRAINT FK_Orders_Customers
        FOREIGN KEY (CustomerID) REFERENCES Customers(CustomerID)
        を追加してください。
```

### バグ2: OrderDetailsのSubtotalカラムがINT型

**該当仕様**: テーブル設計「3.4 OrderDetails」
> Subtotal: **DECIMAL(12,2)** NOT NULL  小計 = Quantity x UnitPrice

また、非機能要件:
> **金額系カラムはすべてDECIMAL型を使用すること（INT不可）**

**該当コード**: `CreateTables.sql` OrderDetailsテーブル定義

```sql
-- 現在の実装（バグ）
CREATE TABLE OrderDetails (
    DetailID    INT          PRIMARY KEY IDENTITY(1,1),
    OrderID     INT          NOT NULL,
    ProductID   INT          NOT NULL,
    Quantity    INT          NOT NULL CHECK (Quantity > 0),
    UnitPrice   DECIMAL(10,2) NOT NULL,
    Subtotal    INT          NOT NULL,   -- ← INT型になっている
    ...
);
```

**修正コード**:
```sql
    Subtotal    DECIMAL(12,2) NOT NULL,
```

**レビュー指摘例**:
```
【指摘箇所】CreateTables.sql: OrderDetailsテーブル Subtotalカラム
【分類】仕様不一致
【重要度】高
【内容】設計仕様書 テーブル設計3.4 では Subtotal のデータ型は
        DECIMAL(12,2) と定義されているが、実装では INT 型になっている。
        Subtotal = Quantity x UnitPrice であり、UnitPriceがDECIMAL(10,2)のため
        小数を含む結果が発生する（例: 2500.50 x 3 = 7501.50）。
        INT型では小数部分が切り捨てられ、金額の不整合が発生する。
        また、非機能要件にて「金額系カラムはすべてDECIMAL型」と明記されている。
【修正案】SubtotalのデータをINTからDECIMAL(12,2)に変更してください。
```

### バグ3: Products.StockQuantityにCHECK制約がない

**該当仕様**: テーブル設計「3.2 Products」
> StockQuantity: INT NOT NULL, **CHECK (StockQuantity >= 0)** 在庫数（0以上）

また、BR-003:
> **在庫数が注文数量を下回る場合はエラーとする**

**該当コード**: `CreateTables.sql` Productsテーブル定義

```sql
-- 現在の実装（バグ）
CREATE TABLE Products (
    ProductID      INT            PRIMARY KEY IDENTITY(1,1),
    ProductName    NVARCHAR(200)  NOT NULL,
    Category       NVARCHAR(50)   NOT NULL,
    UnitPrice      DECIMAL(10,2)  NOT NULL,
    StockQuantity  INT            NOT NULL
    -- ← CHECK制約がない
);
```

**修正コード**:
```sql
CREATE TABLE Products (
    ProductID      INT            PRIMARY KEY IDENTITY(1,1),
    ProductName    NVARCHAR(200)  NOT NULL,
    Category       NVARCHAR(50)   NOT NULL,
    UnitPrice      DECIMAL(10,2)  NOT NULL,
    StockQuantity  INT            NOT NULL CHECK (StockQuantity >= 0)
);
```

**レビュー指摘例**:
```
【指摘箇所】CreateTables.sql: ProductsテーブルStockQuantityカラム
【分類】仕様不一致
【重要度】高
【内容】設計仕様書 テーブル設計3.2 では StockQuantity に
        CHECK (StockQuantity >= 0) 制約を設定すると定義されているが、
        実装ではCHECK制約が定義されていない。
        マイナスの在庫数がデータベースレベルで入力可能な状態であり、
        BR-003（在庫チェック）の前提が崩れる。
        アプリケーション側のバグやSQL直接実行でマイナス在庫になるリスクがある。
【修正案】StockQuantityカラムに CHECK (StockQuantity >= 0) 制約を追加してください。
```

## バグの見つけ方のコツ

1. **仕様書のテーブル定義の「制約」列を1つずつ照合する**
   - PRIMARY KEY、FOREIGN KEY、NOT NULL、CHECK、DEFAULTを漏れなく確認
   - 今回のバグ1とバグ3はこの照合で発見できる

2. **データ型を仕様書と突き合わせる**
   - 特に金額系カラムのDECIMAL指定は厳密に確認
   - INT/DECIMAL/NVARCHARなどの型の違いは見落としやすい
   - 今回のバグ2はこの照合で発見できる

3. **外部キー制約の有無は特に重要**
   - 外部キー制約がないと、存在しないIDの参照が可能になる
   - テーブル間の参照整合性はデータベースの根幹

4. **CHECK制約は「最後の砦」**
   - アプリケーション側でバリデーションしていても、DB直接操作で不正値が入る可能性がある
   - CHECK制約でデータベースレベルでの防御を行うことが重要
