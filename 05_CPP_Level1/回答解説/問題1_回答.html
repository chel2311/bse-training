バグ1: 円の面積計算が間違っている

該当仕様: 3.3 Circle クラス / FR-003

面積 = pi x r x r（pi = 3.14159265358979）

該当コード: Circle.cpp 26行目付近

// 現在の実装（バグ）
double Circle::GetArea()
{
    return PI * radius * 2;
    // → π × r × 2 になっている。2乗ではなく×2。
}

何が問題か:
- 仕様では面積 = pi x r x r（rの2乗）だが、実装は pi x r x 2（rに2を掛けている）
- 半径5.0の場合:
- 正しい面積: 3.14159 x 5 x 5 = 78.54
- バグの面積: 3.14159 x 5 x 2 = 31.42
- 偶然にも GetPerimeter() の計算式 2 * PI * radius と数値が同じになる

半径5.0の円の面積計算

修正コード:

double Circle::GetArea()
{
    return PI * radius * radius;  // π × r × r（rの2乗）
}

レビュー指摘例:

【指摘箇所】Circle.cpp: GetArea() メソッド内
【分類】バグ
【重要度】高
【内容】設計仕様書 3.3 では円の面積は「π × r × r」と定義されているが、
        実装では「PI * radius * 2」となっており、rの2乗ではなく×2になっている。
        すべての円の面積計算が誤った値を返すため、面積合計も不正になる。
【修正案】Circle.cpp の GetArea() 内の計算式を
        「return PI * radius * radius;」に修正してください。

バグ2: ShapeManagerのデストラクタでメモリリークが発生する

該当仕様: FR-006 メモリ管理

new で生成した図形オブジェクトは、ShapeManager のデストラクタで確実に delete すること

該当コード: ShapeManager.cpp 11行目付近

// 現在の実装（バグ）
ShapeManager::~ShapeManager()
{
    // ← shapes vector内の図形ポインタをdeleteする処理がない
    //    vectorは自身が破棄されるが、中のポインタが指すオブジェクトは解放されない
}

何が問題か:
- ShapeManager が破棄されると std::vector 自体は解放されるが、vector内に格納されたポインタが指す先のオブジェクト（new で確保した Circle, Rectangle, Triangle）は delete されない
- C#ではガベージコレクタが自動で回収するが、C++では明示的に delete しないとメモリリークが発生する
- 長時間実行されるCADアプリでは、図形の作成・削除を繰り返すたびにメモリが増え続け、最終的にアプリがクラッシュする可能性がある

C++ メモリライフサイクル（new / delete）
[NG] deleteなし
new Circle(5) → メモリ確保 new Rectangle(3,4) → メモリ確保 new Triangle(...) → メモリ確保 ~ShapeManager() → 何もしない → 3つのメモリが解放されない → メモリリーク発生
[OK] deleteあり
new Circle(5) → メモリ確保 new Rectangle(3,4) → メモリ確保 new Triangle(...) → メモリ確保 ~ShapeManager() { for each: delete shape; } → 全メモリ解放。リークなし

修正コード:

ShapeManager::~ShapeManager()
{
    for (Shape* shape : shapes)
    {
        delete shape;
    }
    shapes.clear();
}

レビュー指摘例:

【指摘箇所】ShapeManager.cpp: デストラクタ（~ShapeManager）
【分類】バグ
【重要度】高
【内容】設計仕様書 FR-006 では「ShapeManagerのデストラクタで保持する全図形の
        メモリを解放（delete）すること」と定義されているが、
        デストラクタの実装が空であり、newで確保した図形オブジェクトが
        deleteされていない。メモリリークが発生する。
【修正案】デストラクタ内でshapes vectorの全要素に対してdeleteを実行し、
        その後shapes.clear()を呼び出してください。

バグ3: 三角形の3辺のバリデーションが不足している

該当仕様: 3.5 Triangle クラス / FR-005

底辺（base_len）、高さ（height）、3辺（a, b, c）はすべて正の値（> 0）であること

該当コード: Triangle.cpp 10行目付近

// 現在の実装（バグ）
Triangle::Triangle(double base_len, double height, double a, double b, double c)
{
    if (base_len <= 0 || height <= 0)  // base_lenとheightだけチェック
    {
        throw std::invalid_argument("底辺と高さは正の値である必要があります");
    }
    // ← a, b, c のチェックがない
    // → マイナスの辺の長さ（例: a=-3.0）でも三角形が作成できてしまう

何が問題か:
- 仕様では底辺・高さに加えて3辺（a, b, c）もすべて正の値であることを要求している
- 実装では base_len と height しかチェックしていない
- side_a = -3.0 のような不正な値で三角形が作成でき、周長計算が不正な値（例: 負の値）を返す可能性がある
- CADソフトで負の辺の長さは物理的にありえないため、不正データがシステムに混入する

修正コード:

Triangle::Triangle(double base_len, double height, double a, double b, double c)
{
    if (base_len <= 0 || height <= 0 || a <= 0 || b <= 0 || c <= 0)
    {
        throw std::invalid_argument("底辺、高さ、3辺はすべて正の値である必要があります");
    }
    // ...
}
[NG] base_lenとheightのみチェック
if (base_len <= 0 || height <= 0) // a, b, c は未チェック // Triangle(3, 4, -1, -2, -3) が通る
[OK] 全パラメータをチェック
if (base_len <= 0 || height <= 0 || a <= 0 || b <= 0 || c <= 0) // すべて正の値を検証

レビュー指摘例:

【指摘箇所】Triangle.cpp: コンストラクタ内のバリデーション処理
【分類】仕様不一致
【重要度】中
【内容】設計仕様書 FR-005 / 3.5 では「底辺、高さ、3辺（a, b, c）はすべて正の値
        （> 0）であること」と定義されているが、実装ではbase_lenとheightの
        チェックのみで、3辺（a, b, c）の正値チェックが行われていない。
        マイナスの辺の長さで三角形が作成可能な状態。
【修正案】if文の条件に「a <= 0 || b <= 0 || c <= 0」を追加し、
        エラーメッセージも「底辺、高さ、3辺はすべて正の値である必要があります」
        に変更してください。

バグの見つけ方のコツ

  1. 仕様書の太字や強調部分は特に注意して確認する
    - 今回のバグ1は計算式の確認、バグ2・3は仕様書で太字になっていた箇所

  2. C++特有のポイントに注目する
    - new があれば対応する delete を探す（バグ2）
    - デストラクタの実装を必ず確認する
    - C#にはないメモリ管理の問題を意識する

  3. 計算式は仕様書の数式と1文字ずつ照合する
    - radius * 2 と radius * radius は見た目が似ているが意味が全く異なる（バグ1）
    - 特に数学の公式を使う計算は、コピーミスが起きやすい

  4. バリデーションは「全パラメータ」が対象か確認する
    - 一部のパラメータだけチェックしている場合がある（バグ3）
    - 仕様書のバリデーション要件を1つずつ照合する

  5. 実際に動かして値を確認する
    - バグ1: 半径5の円の面積が78.54ではなく31.42になる
    - バグ3: Triangle(3, 4, -1, -2, -3) でエラーが出ないことを確認
