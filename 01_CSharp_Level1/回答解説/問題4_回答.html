バグ1: SQLインジェクション脆弱性

該当仕様: 設計仕様書 3.4 SQL実装規約

SQLはパラメータ化クエリを使用すること（SQLインジェクション対策）
文字列結合によるSQL構築は禁止
SqlCommand.Parameters.AddWithValue() でパラメータを設定する

該当コード: DatabaseHelper.cs AddCustomer メソッド 69行目付近

// 現在の実装（バグ）
string sql = "INSERT INTO Customers (Name, Phone, Age, Email) VALUES ('" + name + "', '" + phone + "', " + age + ", '" + email + "')";
SqlCommand cmd = new SqlCommand(sql, conn);
cmd.ExecuteNonQuery();
// → 文字列結合でSQLを構築している。パラメータ化されていない
// → name に「'; DROP TABLE Customers; --」のような値を入力されると
//    テーブルが削除される（SQLインジェクション攻撃）

修正コード:

string sql = "INSERT INTO Customers (Name, Phone, Age, Email) VALUES (@Name, @Phone, @Age, @Email)";
using (SqlCommand cmd = new SqlCommand(sql, conn))
{
    cmd.Parameters.AddWithValue("@Name", name);
    cmd.Parameters.AddWithValue("@Phone", (object?)phone ?? DBNull.Value);
    cmd.Parameters.AddWithValue("@Age", age);
    cmd.Parameters.AddWithValue("@Email", (object?)email ?? DBNull.Value);

    cmd.ExecuteNonQuery();
}
SQLインジェクション攻撃の流れ
ユーザー入力
'; DROP TABLE Customers; --
→
文字列結合でSQL構築
INSERT ... VALUES('入力値')
→
テーブル削除実行
ユーザー入力
田中太郎
→
パラメータ化クエリ
INSERT ... VALUES(@Name)
→
安全にデータ挿入

レビュー指摘例:

【指摘箇所】DatabaseHelper.cs: AddCustomer メソッド内 INSERT文の構築処理
【分類】セキュリティ
【重要度】高
【内容】設計仕様書 3.4「SQL実装規約」では「SQLはパラメータ化クエリを使用すること」
        「文字列結合によるSQL構築は禁止」と定義されているが、
        実装ではINSERT文を文字列結合で構築しており、パラメータ化されていない。
        SQLインジェクション攻撃により、データの漏洩・改ざん・削除が可能な状態。
        同クラスの UpdateCustomer, DeleteCustomer, SearchCustomers メソッドでは
        正しくパラメータ化されているため、AddCustomer のみ対応漏れと思われる。
【修正案】INSERT文のVALUES句をプレースホルダ（@Name, @Phone, @Age, @Email）に変更し、
        SqlCommand.Parameters.AddWithValue() でパラメータを設定してください。

バグ2: DB接続が閉じられない（usingなし）

該当仕様: 設計仕様書 3.4 SQL実装規約

SqlConnection, SqlCommand, SqlDataReader は必ず using で囲むこと
リソースリーク防止のため、IDisposable を実装するオブジェクトは確実に Dispose する

該当コード: DatabaseHelper.cs GetAllCustomers メソッド 27-46行目付近、AddCustomer メソッド 65-70行目付近

// 現在の実装（バグ） - GetAllCustomers
SqlConnection conn = new SqlConnection(connectionString);
conn.Open();

SqlCommand cmd = new SqlCommand("SELECT ...", conn);
SqlDataReader reader = cmd.ExecuteReader();

while (reader.Read())
{
    // ... データ読み取り ...
}
// reader.Close() なし
// conn.Close() なし
// conn.Dispose() なし
// → usingで囲んでいないため、メソッド終了後もDB接続が閉じられない
// → 呼び出すたびに接続が残り続け、接続プールが枯渇する

// 現在の実装（バグ） - AddCustomer も同様
SqlConnection conn = new SqlConnection(connectionString);
conn.Open();
// ... 処理 ...
// → Close/Dispose なし

修正コード:

public List<Customer> GetAllCustomers()
{
    var customers = new List<Customer>();

    using (SqlConnection conn = new SqlConnection(connectionString))
    {
        conn.Open();

        using (SqlCommand cmd = new SqlCommand("SELECT Id, Name, Phone, Age, Email FROM Customers", conn))
        using (SqlDataReader reader = cmd.ExecuteReader())
        {
            while (reader.Read())
            {
                var customer = new Customer(
                    reader.GetInt32(0),
                    reader.GetString(1),
                    reader.IsDBNull(2) ? "" : reader.GetString(2),
                    reader.GetInt32(3),
                    reader.IsDBNull(4) ? "" : reader.GetString(4)
                );
                customers.Add(customer);
            }
        }
    }

    return customers;
}
usingの有無によるDB接続のライフサイクル
[NG] usingなし
SqlConnection conn = new ... conn.Open(); // ... 処理 ... // Close/Dispose なし // → 接続が残り続ける // → 100回呼ぶと100接続が残る // → 接続プール枯渇でDB停止
[OK] usingあり
using (var conn = new ...) { conn.Open(); // ... 処理 ... } // ← 自動でDispose // → 接続が確実に返却される // → 接続プールは安定動作

レビュー指摘例:

【指摘箇所】DatabaseHelper.cs: GetAllCustomers メソッド および AddCustomer メソッド
【分類】バグ
【重要度】高
【内容】設計仕様書 3.4「SQL実装規約」では「SqlConnection, SqlCommand, SqlDataReader は
        必ず using で囲むこと」と定義されているが、
        GetAllCustomers と AddCustomer の2メソッドでは using を使用せずに
        SqlConnection, SqlCommand, SqlDataReader を生成しており、
        Close() も Dispose() も呼ばれていない。
        アプリケーション使用中にDB接続が蓄積し、接続プール（既定100接続）が
        枯渇するとDB操作が一切できなくなる。
        UpdateCustomer, DeleteCustomer, SearchCustomers では正しく using が
        使用されているため、この2メソッドのみ対応漏れと思われる。
【修正案】SqlConnection, SqlCommand, SqlDataReader のすべてを using 文で囲み、
        スコープ終了時に確実に Dispose されるよう修正してください。

バグ3: 例外処理なし（try-catchなし）

該当仕様: 設計仕様書 3.4 SQL実装規約

DB接続時は必ず try-catch で例外処理を行い、ユーザーにエラーメッセージを表示すること
SqlException をキャッチし、ユーザーにわかりやすいメッセージを表示する
アプリケーションがクラッシュしないよう例外を適切にハンドリングする

設計仕様書 7. 非機能要件

DB接続エラー時は適切なメッセージを表示し、アプリがクラッシュしないこと

該当コード: DatabaseHelper.cs GetAllCustomers メソッド 27-46行目、AddCustomer メソッド 63-70行目

// 現在の実装（バグ） - GetAllCustomers
SqlConnection conn = new SqlConnection(connectionString);
conn.Open();                  // ← DB接続失敗で SqlException がスローされる
// ...
SqlDataReader reader = cmd.ExecuteReader();  // ← SQL実行エラーで SqlException がスローされる
// → try-catch が一切ないため、例外がそのまま呼び出し元に伝搬する

// 現在の実装（バグ） - AddCustomer
SqlConnection conn = new SqlConnection(connectionString);
conn.Open();                  // ← try-catch なし
cmd.ExecuteNonQuery();        // ← try-catch なし
// → SQL Server が停止している場合やネットワーク障害時にアプリがクラッシュする

注記: FormMain.cs 側ではDBHelper呼び出しを try-catch で囲んでいるため、
実際にはアプリの即座のクラッシュは防がれています。しかし、設計仕様書では
DatabaseHelper クラス自体に try-catch を実装する方針が示されています。
DatabaseHelper 側でも適切な例外処理を行うことで、より堅牢な設計になります。

修正コード:

public List<Customer> GetAllCustomers()
{
    var customers = new List<Customer>();

    try
    {
        using (SqlConnection conn = new SqlConnection(connectionString))
        {
            conn.Open();

            using (SqlCommand cmd = new SqlCommand("SELECT Id, Name, Phone, Age, Email FROM Customers", conn))
            using (SqlDataReader reader = cmd.ExecuteReader())
            {
                while (reader.Read())
                {
                    var customer = new Customer(
                        reader.GetInt32(0),
                        reader.GetString(1),
                        reader.IsDBNull(2) ? "" : reader.GetString(2),
                        reader.GetInt32(3),
                        reader.IsDBNull(4) ? "" : reader.GetString(4)
                    );
                    customers.Add(customer);
                }
            }
        }
    }
    catch (SqlException ex)
    {
        MessageBox.Show($"データベースエラー: {ex.Message}", "エラー", MessageBoxButtons.OK, MessageBoxIcon.Error);
    }

    return customers;
}

public void AddCustomer(string name, string phone, int age, string email)
{
    try
    {
        using (SqlConnection conn = new SqlConnection(connectionString))
        {
            conn.Open();

            string sql = "INSERT INTO Customers (Name, Phone, Age, Email) VALUES (@Name, @Phone, @Age, @Email)";
            using (SqlCommand cmd = new SqlCommand(sql, conn))
            {
                cmd.Parameters.AddWithValue("@Name", name);
                cmd.Parameters.AddWithValue("@Phone", (object?)phone ?? DBNull.Value);
                cmd.Parameters.AddWithValue("@Age", age);
                cmd.Parameters.AddWithValue("@Email", (object?)email ?? DBNull.Value);

                cmd.ExecuteNonQuery();
            }
        }
    }
    catch (SqlException ex)
    {
        MessageBox.Show($"データベースエラー: {ex.Message}", "エラー", MessageBoxButtons.OK, MessageBoxIcon.Error);
    }
}
try-catch有無のエラー時の動作
[NG] try-catchなし
conn.Open(); // DB停止時に例外 → SqlException が未処理で伝搬 → アプリがクラッシュ → ユーザーの作業が失われる
[OK] try-catchあり
try { conn.Open(); } catch (SqlException ex) { MessageBox.Show("DBエラー..."); } → エラーメッセージを表示 → アプリは継続動作

レビュー指摘例:

【指摘箇所】DatabaseHelper.cs: GetAllCustomers メソッド および AddCustomer メソッド
【分類】仕様不一致
【重要度】中
【内容】設計仕様書 3.4「SQL実装規約」では「DB接続時は必ず try-catch で例外処理を行い、
        ユーザーにエラーメッセージを表示すること」と定義されているが、
        GetAllCustomers と AddCustomer の2メソッドでは try-catch が実装されていない。
        DB接続失敗やSQL実行エラー時に SqlException が未処理のまま伝搬する。
        現状は呼び出し元の FormMain.cs で例外をキャッチしているが、
        設計方針としては DatabaseHelper 側でも例外を処理すべきである。
        UpdateCustomer, DeleteCustomer, SearchCustomers との一貫性もない。
【修正案】GetAllCustomers と AddCustomer の DB操作処理全体を try-catch(SqlException) で
        囲み、エラー時は MessageBox.Show でユーザーにエラーメッセージを表示してください。

バグの見つけ方のコツ

  1. 設計仕様書の「SQL実装規約」を熟読する
    - 今回のバグ3件はすべて設計仕様書 3.4 に太字で記載されたルールへの違反
    - 「禁止」「必ず」「すること」と書かれた項目は特に重点チェック対象

  2. メソッド間の一貫性をチェックする
    - 同じクラス内でも、あるメソッドでは正しく実装されているのに別のメソッドでは漏れている
    - UpdateCustomer / DeleteCustomer / SearchCustomers は正しいのに GetAllCustomers / AddCustomer だけバグがある
    - このパターンは実務でも非常に多い（コピー&ペーストの修正漏れ）

  3. セキュリティ観点を忘れない
    - BSEレビューでは機能面のバグだけでなく、セキュリティ脆弱性の検出も重要
    - SQLインジェクションはOWASP Top 10に含まれる代表的な脆弱性
    - 文字列結合でSQLを構築しているコードは即座にNGと判断できるようにする

  4. リソース管理パターンを確認する
    - C# では IDisposable を実装するオブジェクト（DB接続、ファイルハンドル等）は using 必須
    - using がないコードを見つけたらリソースリークの可能性を疑う

メソッド間の一貫性チェック
using あり（正常）
○ UpdateCustomer
○ DeleteCustomer
○ SearchCustomers
using なし（バグ）
× GetAllCustomers
× AddCustomer

実務での影響度

バグ 影響 発生タイミング
SQLインジェクション データ漏洩・改ざん・削除。最悪の場合サーバー乗っ取り 悪意あるユーザーが入力した瞬間
