バグ1: 税率の適用ミス（常に8%で計算される）

該当仕様: FR-003 小計計算

税率は選択された値を使用する
- 「10%（標準）」選択時: 税率 = 0.10
- 「8%（軽減税率）」選択時: 税率 = 0.08

該当コード: FormMain.cs GetSelectedTaxRate メソッド（113行目付近）

// 現在の実装（バグ）
private decimal GetSelectedTaxRate()
{
    if (cmbTaxRate.SelectedIndex == 0)
    {
        // 「10%（標準）」が選択されている場合
        return 0.08m;  // ← ここが間違い。0.10m であるべき
    }
    else
    {
        // 「8%（軽減税率）」が選択されている場合
        return 0.08m;
    }
}
// → どちらを選んでも常に8%が返される

修正コード:

private decimal GetSelectedTaxRate()
{
    if (cmbTaxRate.SelectedIndex == 0)
    {
        // 「10%（標準）」が選択されている場合
        return 0.10m;
    }
    else
    {
        // 「8%（軽減税率）」が選択されている場合
        return 0.08m;
    }
}

レビュー指摘例:

【指摘箇所】FormMain.cs: GetSelectedTaxRate メソッド内 税率返却処理
【分類】バグ
【重要度】高
【内容】設計仕様書 FR-003 では「10%（標準）」選択時に税率0.10を返すと定義されているが、
        実装ではSelectedIndex == 0（10%（標準）選択時）の分岐で0.08mを返しており、
        どの税率を選択しても常に8%で計算されてしまう。
        標準税率の商品すべてで金額が過少計算となり、売上に影響する重大なバグ。
【修正案】SelectedIndex == 0 の場合のreturn値を 0.08m から 0.10m に変更してください。

バグ2: 合計金額が最後の注文の小計のみ表示される

該当仕様: FR-004 合計計算

全注文の小計の合計を計算して表示する

該当コード: FormMain.cs UpdateTotal メソッド（133行目付近）

// 現在の実装（バグ）
private void UpdateTotal()
{
    int total = 0;

    for (int i = 0; i < orders.Count; i++)
    {
        total = orders[i].Subtotal;  // ← += であるべきところが = になっている
    }

    lblTotal.Text = $"合計金額（税込）: {total:#,0}円";
}
// → ループの各反復でtotalが上書きされ、最後の注文の小計だけが表示される

修正コード:

private void UpdateTotal()
{
    int total = 0;

    for (int i = 0; i < orders.Count; i++)
    {
        total += orders[i].Subtotal;
    }

    lblTotal.Text = $"合計金額（税込）: {total:#,0}円";
}

レビュー指摘例:

【指摘箇所】FormMain.cs: UpdateTotal メソッド内 合計金額計算処理
【分類】バグ
【重要度】高
【内容】設計仕様書 FR-004 では「全注文の小計の合計を計算して表示する」と定義されているが、
        実装ではforループ内の代入演算子が += ではなく = になっているため、
        ループの各反復でtotalが上書きされ、最後の注文の小計のみが合計として表示される。
        注文が複数ある場合に合計金額が正しく計算されない。
【修正案】forループ内の total = orders[i].Subtotal を total += orders[i].Subtotal に
        変更してください。

バグ3: 数量0でも注文可能

該当仕様: FR-005 入力バリデーション

数量: 必須。1以上100以下の整数。不正な場合はエラー「数量は1～100の整数で入力してください」

該当コード: FormMain.cs btnAdd_Click メソッド内（57行目付近）

// 現在の実装（バグ）
if (!int.TryParse(txtQuantity.Text, out int quantity) || quantity < 0 || quantity > 100)
{
    MessageBox.Show("数量は1～100の整数で入力してください", "入力エラー",
        MessageBoxButtons.OK, MessageBoxIcon.Warning);
    return;
}
// → quantity < 0 のため、数量0が有効な値として通ってしまう
// → 小計0円のゴミデータが注文一覧に追加される

修正コード:

if (!int.TryParse(txtQuantity.Text, out int quantity) || quantity < 1 || quantity > 100)
{
    MessageBox.Show("数量は1～100の整数で入力してください", "入力エラー",
        MessageBoxButtons.OK, MessageBoxIcon.Warning);
    return;
}

レビュー指摘例:

【指摘箇所】FormMain.cs: btnAdd_Click メソッド内 数量バリデーション処理
【分類】仕様不一致
【重要度】中
【内容】設計仕様書 FR-005 では数量は「1以上100以下の整数」と定義されているが、
        実装では下限チェックが quantity < 0 となっており、
        数量0の注文が有効な値として追加できてしまう。
        数量0の注文が追加されると小計0円のデータが一覧に残り、
        データの整合性に問題が生じる。
【修正案】数量の下限チェックを quantity < 0 から quantity < 1 に変更してください。

バグの見つけ方のコツ

  1. 計算ロジックは必ず手計算で検算する
    - 単価200 x 数量5 x 1.10 = 1,100円 のように、仕様書の計算式に具体値を入れて検算
    - バグ1は「税率10%なのに1,080円になる」ことで発覚する

  2. 境界値テストを意識する
    - バグ3は数量に「0」を入力することで発覚する
    - 仕様書に書かれた範囲の最小値・最大値は必ずテストする

  3. 複数データで動作確認する
    - バグ2は注文が1件だけでは気づけない
    - 2件以上追加して合計金額が正しいか確認する

  4. 代入演算子（= と +=）の違いに注意する
    - = と += は見た目が似ているがまったく異なる動作をする
    - 集計処理では特に注意が必要
