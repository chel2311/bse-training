演習3 回答・解説: コンフリクト解消

コンフリクトの原因

なぜコンフリクトが発生したか

mainブランチとfeatureブランチで 同じメソッド（Divide）を異なる内容に変更 したため。

ブランチ 変更内容
main エラーメッセージを詳細な日本語に変更（DivideByZeroException → ArgumentException）
feature/divide-validation 整数除算で余りが出る場合の警告メッセージを追加

Gitは「どちらの変更を採用すべきか」を自動的に判断できないため、人間の判断が必要になる。

コンフリクト解消の手順

Step 1: コンフリクトマーカーを理解する

<<<<<<< HEAD
// ここから: 現在のブランチ（main）の変更
// ...
=======
// ここから: マージしようとしたブランチ（feature）の変更
// ...
>>>>>>> feature/divide-validation
マーカー 意味
<<<<<<< HEAD 現在のブランチの変更の開始
======= 2つの変更の区切り
>>>>>>> ブランチ名 マージしようとしたブランチの変更の終了

Step 2: 両方の変更内容を比較する

mainブランチの変更（開発者A）:

public int Divide(int a, int b)
{
    if (b == 0)
    {
        throw new ArgumentException("エラー: 除数に0は指定できません。正の整数を入力してください。");
    }
    return a / b;
}
  • 変更点: エラーメッセージを詳細化
  • 意図: ユーザーにわかりやすいエラーメッセージにする
  • 採用すべきか: はい（ユーザビリティの向上）

featureブランチの変更（開発者B）:

public int Divide(int a, int b)
{
    if (b == 0)
        throw new DivideByZeroException("0で割ることはできません");

    if (a % b != 0)
    {
        Console.WriteLine($"警告: {a} / {b} の結果は切り捨てられます");
    }
    return a / b;
}
  • 変更点: 整数除算の切り捨て警告を追加
  • 意図: 開発者が意図しない切り捨てに気付けるようにする
  • 採用すべきか: はい（品質向上のための警告機能）

Step 3: 最終的なコードを決定する

両方の変更を統合する場合の判断:

  1. エラーメッセージの詳細化 → 採用
  2. 例外の型（ArgumentException vs DivideByZeroException）→ mainの ArgumentException を採用
  3. 整数除算の警告 → 採用

Step 4: 解消後のコード

/// <summary>
/// 除算（エラーメッセージ統一 + 引数チェック強化）
/// </summary>
public int Divide(int a, int b)
{
    if (b == 0)
    {
        throw new ArgumentException("エラー: 除数に0は指定できません。正の整数を入力してください。");
    }

    // 結果が整数にならない場合の警告
    if (a % b != 0)
    {
        Console.WriteLine($"警告: {a} / {b} の結果は切り捨てられます");
    }
    return a / b;
}

コンフリクト解消の3つのパターン

実際のプロジェクトでは、以下のいずれかの判断をする:

パターン1: 自分の変更（HEAD）を採用

相手の変更が不要な場合。
→ <<<<<<< HEAD 以下のコードだけ残し、======= 以下を削除

パターン2: 相手の変更を採用

自分の変更が不要な場合。
→ ======= 以下のコードだけ残し、<<<<<<< HEAD 以下を削除

パターン3: 両方を統合（今回の例）

両方の変更を取り入れる場合。
→ マーカーを削除し、両方の変更を手動で組み合わせる

BSEとしてのコンフリクト解消の注意点

  1. 仕様書を確認する: どちらの変更が仕様に合っているか判断する
  2. 両方の開発者に確認する: 判断に迷う場合は、変更した本人に意図を確認する
  3. テストする: 解消後のコードが正しく動作するか必ずテストする
  4. コミットメッセージに記録する: 何をどう判断して解消したかを記録する

よくある間違い

間違い1: マーカーを残したままコミットする

// これはNG: コンパイルエラーになる
<<<<<<< HEAD
throw new ArgumentException(...);
=======
throw new DivideByZeroException(...);
>>>>>>> feature/divide-validation

→ マーカー（<<<<<<<、=======、>>>>>>>）は必ずすべて削除する。

間違い2: 片方の変更を見落とす

→ ======= の上下両方の変更内容を必ず確認する。

間違い3: 解消せずにマージを中止する

git merge --abort  # マージをなかったことにする

→ 問題の先送りにしかならない。根本的にはコンフリクトを解消する必要がある。
