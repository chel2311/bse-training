Git / Pull Request レビューチェックリスト（BSE向け）

使い方

Pull Requestのレビュー時に以下の項目を1つずつ確認していく。
各項目を確認したら「OK」「NG」「N/A（該当なし）」を記入する。

1. コミットメッセージ

No 確認項目 結果
1-1 コミットメッセージが変更内容を適切に説明しているか
1-2 日本語（または規約で定められた言語）で書かれているか
1-3 1つのコミットに1つの変更内容になっているか（複数の無関係な変更が混在していないか）
1-4 「修正」「変更」だけでなく「何を」「なぜ」が書かれているか

コミットメッセージの書き方

良い例:

顧客追加時の年齢バリデーションを追加（0-150の範囲チェック）
CSVエクスポート機能でUTF-8 BOM付きエンコーディングに変更
削除処理に確認ダイアログを追加（設計仕様書FR-003対応）

悪い例:

修正                          ← 何を修正したか不明
バグ修正                      ← どのバグか不明
WIP                          ← 作業途中のコミット
あああ                        ← 意味不明
FormMain.csを変更した          ← ファイル名だけでは内容がわからない

2. Pull Requestの内容

No 確認項目 結果
2-1 PRタイトルが変更内容を簡潔に説明しているか
2-2 PR説明に「何を変更したか」が書かれているか
2-3 PR説明に「なぜ変更したか」（背景・目的）が書かれているか
2-4 関連する課題番号やチケット番号が記載されているか
2-5 マージ先のブランチは正しいか（feature → develop等）
2-6 PRの変更範囲は適切か（大きすぎないか）

3. 差分（Diff）の確認

No 確認項目 結果
3-1 変更内容は設計仕様書の要件を満たしているか
3-2 不要なファイルの変更が含まれていないか（空白の変更、自動フォーマット等）
3-3 デバッグ用コード（Console.WriteLine、コメントアウト等）が残っていないか
3-4 ハードコーディングされた値（ファイルパス、URL、パスワード等）がないか
3-5 新規追加されたリソース（ファイル、DB接続等）は適切に解放されているか
3-6 エラー処理（try-catch）は適切か
3-7 入力バリデーションは仕様通りか

4. diffの読み方ガイド

基本記号

記号 意味 GitHub上の色
- 削除された行 赤背景
+ 追加された行 緑背景
（空白） 変更なし（前後の文脈表示） 白背景
@@ 変更箇所の位置情報（行番号） 紫

diff出力の読み方

diff --git a/FormMain.cs b/FormMain.cs
index abc1234..def5678 100644
--- a/FormMain.cs                    ← 変更前のファイル
+++ b/FormMain.cs                    ← 変更後のファイル
@@ -25,6 +25,11 @@ private void btnAdd_Click  ← 25行目付近、変更前6行→変更後11行
     // バリデーション
     if (string.IsNullOrEmpty(txtName.Text))
     {
-        MessageBox.Show("名前を入力してください");
+        MessageBox.Show("名前を入力してください",
+            "入力エラー",
+            MessageBoxButtons.OK,
+            MessageBoxIcon.Warning);
         return;
     }
+
+    // 年齢の範囲チェックを追加
+    if (!int.TryParse(txtAge.Text, out int age) || age < 0 || age > 150)
+    {
+        MessageBox.Show("年齢は0～150の整数で入力してください");
+        return;
+    }

確認の手順

  1. @@ 行で変更箇所の位置（メソッド名、行番号）を確認する
  2. -（赤）の行で「何が削除されたか」を確認する
  3. +（緑）の行で「何が追加されたか」を確認する
  4. 変更なしの行（白）で前後の文脈を把握する
  5. 変更内容が設計仕様書の要件と一致するか照合する

5. コード品質

No 確認項目 結果
5-1 変数名・メソッド名が処理内容を適切に表しているか
5-2 不要なコメントアウトされたコードが残っていないか
5-3 例外処理が空のcatchブロック（エラーの握りつぶし）になっていないか
5-4 ファイルI/OやDB接続がusing文で適切に管理されているか
5-5 文字エンコーディングが適切に指定されているか（日本語ファイル）

6. セキュリティ

No 確認項目 結果
6-1 SQLにパラメータを使っているか（文字列結合はNG）
6-2 パスワードや接続文字列がソースコードにハードコーディングされていないか
6-3 ファイルパスがハードコーディングされていないか
6-4 機密情報を含むファイル（.config、.env等）がコミットされていないか

7. テスト・動作確認

No 確認項目 結果
7-1 変更した機能が正しく動作するか（テスト実行またはデバッグ確認）
7-2 変更していない既存機能に影響がないか（回帰テスト）
7-3 エラーケース（不正入力、接続エラー等）でも正常に動作するか

指摘テンプレート

PR上でのコメント形式

【指摘箇所】ファイル名: 行番号付近
【分類】バグ / 仕様不一致 / 改善提案 / セキュリティ
【重要度】高 / 中 / 低
【内容】○○の処理で△△となっているが、□□とすべきではないか。
【修正案】△△を□□に変更してください。

指摘の書き方のコツ

  1. 具体的に書く: 「ここがおかしい」ではなく「○行目の△△が仕様と異なる」
  2. 理由を書く: なぜ問題なのかを説明する（バグになる、リソースリークする等）
  3. 修正案を提示する: 問題を指摘するだけでなく、どう修正すべきかを示す
  4. 敬意を持つ: 「ここが間違っている」ではなく「ここは○○とした方が良いのではないか」
