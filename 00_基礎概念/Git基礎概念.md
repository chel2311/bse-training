# Git バージョン管理 基礎概念（BSE向け）

> コードの変更履歴を管理し、Pull Requestをレビューできることが目標

---

## 1. Gitとは

### バージョン管理とは何か

ソースコードの「変更履歴」を記録・管理する仕組み。
誰が、いつ、どのファイルを、どう変更したかを追跡できる。

```
なぜ必要か:
- 「昨日動いていたコードに戻したい」 → 過去のバージョンに戻せる
- 「誰がこのバグを入れたか知りたい」 → 変更履歴を辿れる
- 「複数人で同じファイルを編集したい」 → 変更を統合（マージ）できる
- 「この変更は正しいかレビューしたい」 → Pull Requestで差分を確認できる
```

### Gitの基本用語

| 用語 | 意味 |
|------|------|
| リポジトリ（Repository） | プロジェクト全体の保管庫。変更履歴も含む |
| ローカルリポジトリ | 自分のPCにあるリポジトリ |
| リモートリポジトリ | サーバー上（GitHub、Azure DevOps等）にあるリポジトリ |
| コミット（Commit） | 変更内容を履歴として記録すること。「セーブポイント」のようなもの |
| ブランチ（Branch） | 作業を分岐させる機能。本流に影響せず開発できる |
| マージ（Merge） | 分岐した作業を本流に統合すること |
| プル（Pull） | リモートの最新状態をローカルに取り込む |
| プッシュ（Push） | ローカルの変更をリモートに送る |
| クローン（Clone） | リモートリポジトリを丸ごとローカルにコピーする |
| ステージング（Staging） | コミットする変更を選択する中間領域 |

## 2. 基本コマンド一覧

### リポジトリの取得・確認

```bash
# リモートリポジトリをローカルにコピー（最初に1回だけ実行）
git clone https://github.com/company/project.git

# 現在の状態を確認（変更したファイル、ステージング状態）
git status
```

`git status` の出力例:
```
On branch main
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
        modified:   FormMain.cs        ← 変更したがまだステージングしていない

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        NewFile.cs                     ← 新規ファイル（まだGit管理外）
```

### 変更の記録（コミットの流れ）

```bash
# 1. 変更したファイルをステージング（コミット対象に選択）
git add FormMain.cs

# 全ファイルをまとめてステージング
git add .

# 2. コミット（変更を履歴として記録）
git commit -m "顧客追加時のバリデーションを修正"

# 3. リモートに送信
git push
```

### 最新の取得

```bash
# リモートの最新状態をローカルに反映
git pull
```

**作業前に必ず `git pull` を実行する習慣をつける。**
他の人の変更を取り込まずに作業すると、後でコンフリクトが発生しやすい。

### ブランチ操作

```bash
# ブランチ一覧を表示（*が現在のブランチ）
git branch

# 新しいブランチを作成
git branch feature/csv-export

# ブランチを切り替える
git checkout feature/csv-export

# ブランチの作成と切り替えを同時に行う
git checkout -b feature/csv-export

# mainブランチにfeatureブランチの変更を統合
git checkout main
git merge feature/csv-export
```

### 履歴の確認

```bash
# コミット履歴を表示
git log

# コミット履歴を1行ずつ簡潔に表示
git log --oneline

# 特定ファイルの変更履歴
git log FormMain.cs

# 変更内容の差分を表示
git diff

# ステージング済みの差分を表示
git diff --staged

# 2つのブランチ間の差分
git diff main..feature/csv-export
```

`git log --oneline` の出力例:
```
a3f7b2c 顧客削除時の確認ダイアログを追加
b8e1d4f 年齢バリデーションの範囲チェックを追加
c5a9e3g 顧客追加機能の初期実装
d2b6f1h プロジェクトの初期セットアップ
```

### レビューで使うコマンドまとめ

| よく使う場面 | コマンド |
|-------------|---------|
| 最新のコードを取得したい | `git pull` |
| 誰が何を変更したか確認したい | `git log` |
| 変更内容の詳細を見たい | `git diff` |
| 今の作業状態を確認したい | `git status` |
| 特定のブランチに切り替えたい | `git checkout ブランチ名` |

## 3. ブランチ戦略の基本

チーム開発では、ブランチの使い方にルールを設ける。
以下は最も一般的な3層構造。

```
main（本番環境）
  │
  ├── develop（開発の統合ブランチ）
  │     │
  │     ├── feature/customer-add（顧客追加機能）
  │     ├── feature/csv-export（CSVエクスポート機能）
  │     └── feature/search-fix（検索バグ修正）
  │
  └── hotfix/urgent-fix（緊急バグ修正）
```

| ブランチ | 目的 | マージ先 |
|---------|------|---------|
| main | 本番環境のコード。直接変更しない | - |
| develop | 開発中のコードを統合するブランチ | main |
| feature/* | 個別の機能開発やバグ修正 | develop |
| hotfix/* | 本番の緊急バグ修正 | main, develop |

### ブランチのイメージ図

```
時間の流れ →

main:     o───────o──────────────────────o───── (本番リリース)
          │       ↑                      ↑
develop:  o───o───o───o───o───o───o──────o───── (開発統合)
              │       ↑   │       ↑
feature/A:    o───o───o   │       │        (機能Aの開発)
                          │       │
feature/B:                o───o───o        (機能Bの開発)

o = コミット
│ = ブランチの分岐
↑ = マージ（統合）
```

### BSEが知っておくべきこと

- 開発者は `feature/` ブランチで作業し、`develop` へのマージ前にPull Requestを出す
- BSEはPull Requestでコードレビューを行い、承認（Approve）または差し戻し（Request Changes）する
- `main` への直接コミットは禁止が基本

## 4. コンフリクトとは

2人が**同じファイルの同じ箇所**を別々に変更した場合、Gitはどちらを採用すべきか判断できない。
これが「コンフリクト（競合）」。

### コンフリクトが発生する例

```
開発者A: GetTaxRate メソッドの税率を 0.08 → 0.10 に変更
開発者B: GetTaxRate メソッドの税率を 0.08 → 0.08（軽減税率の分岐を追加）

→ 同じ行を異なる内容に変更 → マージ時にコンフリクト発生
```

### コンフリクト発生時のファイル表示

```csharp
public decimal GetTaxRate()
{
<<<<<<< HEAD
    return 0.10m;  // 開発者Aの変更（現在のブランチ）
=======
    // 軽減税率対応
    if (IsReducedTaxRate)
        return 0.08m;
    return 0.10m;  // 開発者Bの変更（マージしようとしたブランチ）
>>>>>>> feature/reduced-tax
}
```

| マーカー | 意味 |
|---------|------|
| `<<<<<<< HEAD` | 現在のブランチ（自分）の変更の開始 |
| `=======` | 区切り線 |
| `>>>>>>> ブランチ名` | マージしようとしたブランチの変更の終了 |

### コンフリクトの解消手順

1. コンフリクトマーカー（`<<<<<<<`、`=======`、`>>>>>>>`）を削除する
2. 正しいコードになるよう手動で編集する
3. `git add` でステージングし、`git commit` で解消を確定する

## 5. Pull Request（PR）の概念と流れ

Pull Requestとは、「自分の変更を本流ブランチに統合（マージ）してほしい」というリクエスト。
GitHub、Azure DevOps、GitLab等のWebサービス上で行う。

### PRの流れ

```
1. 開発者がfeatureブランチで作業・コミット
2. 開発者がリモートにpush
3. 開発者がPull Requestを作成（変更内容の説明を記載）
4. レビュアー（BSE等）が差分を確認
5. 問題があれば指摘コメントを記入 → 開発者が修正
6. 問題なければ「承認（Approve）」
7. マージ実行 → developブランチに統合
```

### PRに含まれる情報

| 項目 | 内容 |
|------|------|
| タイトル | 変更内容の要約（例: 「CSVエクスポート機能の追加」） |
| 説明 | 何を変更したか、なぜ変更したかの詳細 |
| 差分（Diff） | 変更前と変更後のコードの違い |
| コミット一覧 | このPRに含まれる全コミット |
| レビューコメント | レビュアーの指摘やフィードバック |

### diffの読み方

```diff
@@ -15,7 +15,10 @@ public class CustomerManager
     public void AddCustomer(string name, string phone)
     {
-        // 古いコード（削除された行）
-        customers.Add(new Customer(name, phone));
+        // 新しいコード（追加された行）
+        if (string.IsNullOrEmpty(name))
+            throw new ArgumentException("名前は必須です");
+        customers.Add(new Customer(name, phone));
     }
 }
```

| 記号 | 意味 | 色（GitHub等） |
|------|------|---------------|
| `-` | 削除された行 | 赤 |
| `+` | 追加された行 | 緑 |
| （空白） | 変更なしの行（前後の文脈） | 白 |
| `@@` | 変更箇所の位置情報 | 紫 |

## 6. BSEとしてGitで何をするか

BSEの主な役割は「コードを書く」ことではなく「レビュー」すること。
Gitに関しては以下のスキルが必要。

### 必須スキル

| スキル | 具体的な内容 |
|--------|------------|
| PRレビュー | Pull Requestの差分を読み、問題点を指摘する |
| 変更履歴の確認 | `git log` で誰がいつ何を変更したかを追跡する |
| 差分の読み方 | `git diff` やPR画面で変更内容を正確に理解する |
| ブランチの理解 | どのブランチで何の作業が行われているか把握する |

### PRレビューで確認すること

1. **仕様との整合性**: 変更内容が設計仕様書の要件を満たしているか
2. **既存コードへの影響**: 変更が他の機能を壊していないか
3. **コード品質**: usingの使用、エラー処理、バリデーション等
4. **セキュリティ**: SQLインジェクション、ハードコーディングされた機密情報等
5. **コミットメッセージ**: 変更内容を適切に説明しているか

## 7. Visual StudioでのGit操作

Visual Studioには「Git Changes」ウィンドウが統合されており、コマンドを使わずにGit操作が可能。

### Git Changesウィンドウ（Visual Studio 2019以降）

| 操作 | Visual Studioでの手順 |
|------|---------------------|
| Clone | ファイル → リポジトリのクローン → URLを入力 |
| Status確認 | Git Changes ウィンドウで変更ファイル一覧を確認 |
| Add（ステージング） | 変更ファイルの横の「+」ボタンをクリック |
| Commit | メッセージを入力して「コミット」ボタン |
| Push | Git Changes 上部の「↑ プッシュ」ボタン |
| Pull | Git Changes 上部の「↓ プル」ボタン |
| Branch作成 | 右下のブランチ名をクリック → 「新しいブランチ」 |
| Branch切り替え | 右下のブランチ名をクリック → 切り替え先を選択 |
| 差分の確認 | 変更ファイルをダブルクリック → 差分表示 |
| 履歴の確認 | Git → コミット履歴の表示 |

### チーム エクスプローラー（Visual Studio 2017以前）

古いバージョンのVisual Studioでは「チーム エクスプローラー」を使用する。

| 操作 | 手順 |
|------|------|
| 表示 | 表示メニュー → チーム エクスプローラー |
| Clone | 接続 → リポジトリのクローン |
| 変更の確認 | ホーム → 変更 |
| ブランチ操作 | ホーム → ブランチ |
| 同期（Push/Pull） | ホーム → 同期 |

## 8. Gitの作業フロー図

### 日常の作業フロー

```
[リモートリポジトリ]
       │
       │ git clone（初回のみ）
       ↓
[ローカルリポジトリ]
       │
       │ git pull（作業開始前に最新化）
       ↓
[ファイルを編集]
       │
       │ git add（変更をステージング）
       ↓
[ステージングエリア]
       │
       │ git commit（履歴として記録）
       ↓
[ローカルリポジトリ]
       │
       │ git push（リモートに送信）
       ↓
[リモートリポジトリ]
       │
       │ Pull Request作成
       ↓
[レビュー → 承認 → マージ]
```

### 3つの領域

```
作業ディレクトリ          ステージング           ローカルリポジトリ
（実際のファイル）     （コミット準備領域）      （変更履歴の保管庫）
      │                     │                       │
      │── git add ──→       │                       │
      │                     │── git commit ──→      │
      │                     │                       │── git push ──→ リモート
      │←────────── git pull ─────────────────────────│
```

## レビューで見るポイント（まとめ）

- コミットメッセージは変更内容を正しく説明しているか？
- 1つのコミットに複数の無関係な変更が混在していないか？
- featureブランチ名は機能内容を反映しているか？
- Pull Requestの説明に「何を」「なぜ」変更したかが書かれているか？
- マージ先のブランチは正しいか？（feature → develop、hotfix → main）
- コンフリクトが未解消のままマージされていないか？
