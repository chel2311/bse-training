# コードリーディングガイド（BSE向け）

> FormMain.csを開いたが「何が書いてあるか分からない」状態を脱出するためのガイド

---

## 1. コードを読む前に知っておくこと

### コードは「上から下に順番に読む」ものではない

本や文書は1ページ目から順に読む。しかしWinFormsアプリのコードは違う。

WinFormsアプリは**イベント駆動**で動く。つまり「ユーザーがボタンを押す」「フォームが開く」といった**イベント（出来事）が起きた時に、対応する処理が実行される**仕組みになっている。したがって、コードを読むときは**「どのイベントで、どの処理が動くか」**を追いかける。

```
上から下に読む（間違い）:
  1行目 → 2行目 → 3行目 → ... → 最終行

イベントから追う（正しい）:
  「追加ボタンをクリック」→ btnAdd_Click メソッドを読む
  「削除ボタンをクリック」→ btnDelete_Click メソッドを読む
  「フォームが開かれた」  → FormMain_Load メソッドを読む
```

### メソッド名を見れば何の処理か分かる

WinFormsには命名規則がある。メソッド名は「**コントロール名_イベント名**」の形式になっている。

```
btnAdd_Click       → 追加ボタン（btnAdd）がクリック（Click）された時の処理
btnDelete_Click    → 削除ボタン（btnDelete）がクリック（Click）された時の処理
btnSearch_Click    → 検索ボタン（btnSearch）がクリック（Click）された時の処理
FormMain_Load      → メインフォーム（FormMain）が読み込み（Load）された時の処理
dgvCustomers_SelectionChanged → 一覧（dgvCustomers）の選択が変わった（SelectionChanged）時の処理
```

### コントロール名のプレフィックス（接頭辞）一覧

コントロール名の先頭2〜3文字を見れば、それが画面上のどの部品かが分かる。

| プレフィックス | 正式名称 | 画面上の部品 | 例 |
|-------------|---------|------------|-----|
| btn | Button | ボタン | btnAdd（追加ボタン）、btnDelete（削除ボタン） |
| txt | TextBox | テキスト入力欄 | txtName（氏名入力欄）、txtAge（年齢入力欄） |
| dgv | DataGridView | 一覧表（表形式のデータ表示） | dgvCustomers（顧客一覧） |
| lbl | Label | ラベル（表示専用の文字） | lblStatus（ステータス表示）、lblName（「氏名:」の文字） |
| cmb | ComboBox | ドロップダウン選択リスト | cmbDepartment（部署選択） |
| chk | CheckBox | チェックボックス | chkIsActive（有効フラグ） |
| rdo / rb | RadioButton | ラジオボタン（択一選択） | rdoMale（男性） |
| lst | ListBox | リストボックス | lstItems（項目一覧） |
| mnu | MenuStrip / MenuItem | メニュー | mnuFile（ファイルメニュー） |

### イベント名の一覧

| イベント名 | いつ発生するか |
|-----------|-------------|
| _Click | ボタンなどがクリックされた時 |
| _Load | フォームが開かれて読み込まれた時 |
| _SelectionChanged | 一覧の選択行が変わった時 |
| _TextChanged | テキストボックスの内容が変わった時 |
| _SelectedIndexChanged | コンボボックスやリストの選択が変わった時 |
| _FormClosing | フォームが閉じられようとする時 |
| _CellClick | 一覧のセルがクリックされた時 |
| _KeyDown / _KeyPress | キーボードのキーが押された時 |

### つまり、メソッド名だけで処理の目的が分かる

```
btnDelete_Click           → 「削除ボタンがクリックされた時の処理」
dgvCustomers_CellClick    → 「顧客一覧のセルがクリックされた時の処理」
FormMain_FormClosing      → 「メインフォームが閉じられる時の処理」
cmbDepartment_SelectedIndexChanged → 「部署コンボボックスの選択が変わった時の処理」
```

## 2. コードの読み方 5ステップ

FormMain.csを開いた時、全部読もうとしてはいけない。以下の5ステップで**必要な部分だけ**を読む。

### Step 1: メソッド一覧を見る

まずメソッド名だけを拾い読みして、「このファイルにはどんな処理があるのか」を把握する。

問題1のFormMain.csのメソッド一覧:

```
FormMain_Load                    ← フォーム読み込み時
btnAdd_Click                     ← 追加ボタン
btnEdit_Click                    ← 編集ボタン
btnDelete_Click                  ← 削除ボタン
btnSearch_Click                  ← 検索ボタン
btnShowAll_Click                 ← 全件表示ボタン
dgvCustomers_SelectionChanged    ← 一覧の選択変更時
RefreshGrid                      ← 一覧を更新する内部処理
UpdateStatus                     ← ステータスバーを更新する内部処理
```

この時点で「追加・編集・削除・検索・全件表示の5つのボタン処理がある」と分かる。コードの中身はまだ読まなくてよい。

### Step 2: 仕様書の機能と対応づける

設計仕様書の各機能要件（FR-001, FR-002...）が、どのメソッドに対応するかを整理する。

| 仕様書の機能 | 機能番号 | 対応するメソッド |
|------------|---------|---------------|
| 顧客追加 | FR-001 | btnAdd_Click |
| 顧客編集 | FR-002 | btnEdit_Click |
| 顧客削除 | FR-003 | btnDelete_Click |
| 検索 | FR-004 | btnSearch_Click |
| 入力バリデーション | FR-005 | btnAdd_Click内のチェック処理 |
| ステータス表示 | FR-006 | UpdateStatus |

### Step 3: 該当メソッドだけ読む

バグを探す時は**全部を読む必要はない**。仕様書で気になった機能のメソッドだけを集中して読む。

例えば「追加機能にバグがありそうだ」と思ったら、`btnAdd_Click` だけを読む。他のメソッドは後回しでよい。

### Step 4: 日本語に翻訳しながら読む

1行ずつ「この行は何をしているか」を自分の言葉で日本語にする。

```csharp
// btnAdd_Click メソッドの中身を1行ずつ翻訳する
if (string.IsNullOrWhiteSpace(txtName.Text))        // 「もし氏名欄が空なら」
{
    MessageBox.Show("氏名を入力してください", ...);   // 「エラーメッセージを表示して」
    return;                                           // 「処理を中断する」
}

if (!int.TryParse(txtAge.Text, out int age))         // 「もし年齢欄が整数でなければ」
{
    MessageBox.Show("年齢は整数で入力してください",...); // 「エラーメッセージを表示して」
    return;                                           // 「処理を中断する」
}

var customer = new Customer(nextId++, ...);           // 「新しい顧客データを作って」
customers.Add(customer);                              // 「顧客リストに追加して」
RefreshGrid(customers);                               // 「一覧を更新する」
```

### Step 5: 仕様書と突き合わせる

翻訳した内容を仕様書と1つずつ比較する。「仕様に書いてあるのに実装にない」ものがバグ。

```
仕様FR-005: 「年齢は0以上150以下の整数」
  → 翻訳: 「もし年齢欄が整数でなければ」 ... 範囲チェックがない!
  → バグ: 年齢の範囲チェック（0〜150）が実装されていない

仕様FR-001: 「追加後、入力欄をクリアする」
  → 翻訳: 「リストに追加して」→「一覧を更新する」... クリア処理がない!
  → バグ: 追加後に入力欄をクリアする処理がない
```

## 3. C#コードの「翻訳辞書」（頻出パターン）

FormMain.csに出てくるコードパターンを日本語に翻訳する辞書。コードを読む時にこの辞書を横に置いて参照する。

### 3.1 入力チェック系

```csharp
if (string.IsNullOrWhiteSpace(txtName.Text))
```
**翻訳**: 「もし 氏名テキストボックス が 空欄（または空白だけ） なら」

```csharp
if (!int.TryParse(txtAge.Text, out int age))
```
**翻訳**: 「年齢テキストボックスの文字を整数に変換してみる。変換できなかったら（!がついているので条件が反転）」

```csharp
if (age < 0 || age > 150)
```
**翻訳**: 「もし 年齢が0未満 または 年齢が150超 なら」（`||` は「または」の意味）

```csharp
if (age >= 0 && age <= 150)
```
**翻訳**: 「もし 年齢が0以上 かつ 年齢が150以下 なら」（`&&` は「かつ」の意味）

```csharp
if (string.IsNullOrEmpty(txtEmail.Text))
```
**翻訳**: 「もし メールテキストボックス が 空（null または空文字""） なら」
  - `IsNullOrEmpty`: nullか空文字をチェック
  - `IsNullOrWhiteSpace`: nullか空文字か空白だけをチェック（こちらの方が厳しい）

### 3.2 メッセージ表示系

```csharp
MessageBox.Show("エラーです", "警告", MessageBoxButtons.OK, MessageBoxIcon.Warning)
```
**翻訳**: 「"エラーです"というメッセージを、タイトル"警告"、OKボタン、警告アイコン付きのダイアログで表示する」

```csharp
var result = MessageBox.Show("削除しますか？", "確認",
    MessageBoxButtons.YesNo, MessageBoxIcon.Question);
if (result == DialogResult.Yes)
{
    // 削除処理
}
```
**翻訳**: 「"削除しますか？"と、はい/いいえの確認ダイアログを表示する。はいが押されたら中の処理を実行する」

```csharp
return;
```
**翻訳**: 「ここで処理を中断して戻る（この行より下は実行されない）」

### 3.3 オブジェクト作成・リスト操作系

```csharp
var customer = new Customer(nextId++, txtName.Text, txtPhone.Text, age, txtEmail.Text);
```
**翻訳**: 「新しい顧客オブジェクトを作る。IDは自動採番（nextIdを使って1増やす）、氏名・電話・年齢・メールは入力欄の値を使う」

```csharp
customers.Add(customer);
```
**翻訳**: 「顧客リストに、今作った顧客を追加する」

```csharp
customers.Remove(customer);
```
**翻訳**: 「顧客リストから、指定した顧客を削除する」

```csharp
customers.Find(c => c.Id == selectedId)
```
**翻訳**: 「顧客リストから、IDが selectedId と一致するものを1件探す」
  - `c =>` の部分は「リストの各要素を c という名前で1つずつ見て」という意味
  - 見つからない場合は `null`（何もない）が返る

```csharp
customers.FindAll(c => c.Name.Contains(keyword))
```
**翻訳**: 「顧客リストから、名前にキーワードを含むものを全部探す（結果は複数件）」

```csharp
customers.FindAll(c =>
    c.Name.Contains(keyword, StringComparison.OrdinalIgnoreCase) ||
    c.Email.Contains(keyword, StringComparison.OrdinalIgnoreCase))
```
**翻訳**: 「顧客リストから、名前またはメールにキーワードを含むものを全部探す（大文字小文字を区別しない）」

### 3.4 一覧表（DataGridView）操作系

```csharp
dgvCustomers.DataSource = null;
dgvCustomers.DataSource = list;
```
**翻訳**: 「一覧表のデータを一度クリアして、新しいデータ（list）で表示し直す」
  - 一度 `null` にしないと画面が更新されない場合がある

```csharp
dgvCustomers.SelectedRows.Count == 0
```
**翻訳**: 「一覧で何も行が選択されていない」

```csharp
dgvCustomers.SelectedRows[0].Cells["Name"].Value
```
**翻訳**: 「一覧で選択中の行（1行目）の"Name"列の値を取得する」
  - `[0]` は「最初の」という意味（プログラムでは番号を0から数える）

```csharp
(int)dgvCustomers.SelectedRows[0].Cells["Id"].Value
```
**翻訳**: 「選択行のId列の値を整数（int）として取得する」
  - `(int)` は型変換（キャスト）。値を整数型として扱うよう指定する

### 3.5 テキストボックス操作系

```csharp
txtName.Text = "";
```
**翻訳**: 「氏名テキストボックスを空にする（クリアする）」

```csharp
txtName.Text = row.Cells["Name"].Value?.ToString() ?? "";
```
**翻訳**: 「選択行のName列の値を文字列にして氏名テキストボックスにセットする。値がnullなら空文字にする」
  - `?.` は「もし null でなければ」（null安全呼び出し）
  - `??` は「もし左側が null なら右側の値を使う」（null合体演算子）

### 3.6 計算・集計系

```csharp
displayList.Sum(c => c.Age)
```
**翻訳**: 「リスト内の全顧客の年齢を合計する」

```csharp
(double)totalAge / count
```
**翻訳**: 「合計年齢を小数型に変換してから件数で割る（小数の結果が得られる）」
  - `(double)` をつけないと整数同士の割り算になり、小数部分が切り捨てられる
  - 例: `63 / 2` → `31`（整数割り算）、`(double)63 / 2` → `31.5`（小数割り算）

```csharp
count > 0 ? (double)totalAge / count : 0
```
**翻訳**: 「もし件数が0より大きければ平均を計算する。0件なら結果は0にする」
  - `条件 ? 値A : 値B` は「条件が真なら値A、偽なら値B」の省略記法（三項演算子）
  - 0で割る（ゼロ除算）エラーを防ぐためのガード処理

### 3.7 文字列操作系

```csharp
$"件数: {count}件  合計年齢: {totalAge}"
```
**翻訳**: 「文字列の中に変数の値を埋め込む。{count} の部分が実際の値に置き換わる」
  - 先頭の `$` が目印。文字列補間（String Interpolation）と呼ぶ
  - 例: countが3なら → `"件数: 3件  合計年齢: 105"`

```csharp
$"{avgAge:F1}"
```
**翻訳**: 「avgAgeの値を小数第1位まで表示する」
  - `:F1` は書式指定。F1=小数1桁、F2=小数2桁

```csharp
keyword.Contains("検索文字")
```
**翻訳**: 「keywordの中に"検索文字"が含まれているか」

```csharp
StringComparison.OrdinalIgnoreCase
```
**翻訳**: 「大文字小文字を区別しないで比較する設定」
  - `Contains(keyword)` だけだと大文字小文字を区別する
  - `Contains(keyword, StringComparison.OrdinalIgnoreCase)` で区別しなくなる

### 3.8 クラス・宣言系

```csharp
private List<Customer> customers = new List<Customer>();
```
**翻訳**: 「顧客データのリスト（一覧）を作成する。最初は空のリスト」
  - `List<Customer>` は「Customer型のリスト」
  - `private` は「このクラスの中だけで使える」という意味

```csharp
private int nextId = 1;
```
**翻訳**: 「次に使うIDを保持する変数。初期値は1」

```csharp
public partial class FormMain : Form
```
**翻訳**: 「FormMainというフォームクラスの定義。partialは複数ファイルに分かれていることを示す」
  - FormMain.cs（手書きのコード）とFormMain.Designer.cs（自動生成）に分かれている

```csharp
private void btnAdd_Click(object sender, EventArgs e)
```
**翻訳**: 「追加ボタンがクリックされた時に実行されるメソッド」
  - `private` = このクラスの中だけで使える
  - `void` = 戻り値なし（結果を返さない処理）
  - `object sender` = イベントを発生させたオブジェクト（この場合はボタン）
  - `EventArgs e` = イベントの追加情報（通常は使わない）

### 3.9 LINQ式（ラムダ式を使ったデータ操作）

```csharp
displayList.Select(c => new { c.Id, c.Name, c.Phone, c.Age, c.Email }).ToList()
```

  - `c =>` は「各要素をcという名前で扱って」の意味
  - `.ToList()` は結果をリスト型に変換

```csharp
displayList.Where(c => c.Age >= 20).ToList()
```
**翻訳**: 「リストから年齢が20以上のものだけを抽出してリストにする」

```csharp
displayList.Count
```
**翻訳**: 「リストの件数」

## 4. Designer.csの読み方

### Designer.csとは何か

Visual Studioでフォームにボタンやテキストボックスを配置すると、その情報が**FormMain.Designer.cs**に自動生成される。

- **FormMain.cs**: 自分で書くコード（ボタンをクリックした時の処理など）
- **FormMain.Designer.cs**: VSが自動生成するコード（画面部品の配置・設定）

基本的にDesigner.csは**手で編集しない**。しかしレビューでは**イベントの接続確認**に使う。

### イベント接続とは

ボタンを画面に置いただけでは、クリックしても何も起きない。**「このボタンをクリックしたら、このメソッドを実行する」という接続**がDesigner.csに書かれている。

```csharp
// Designer.cs内の記述
this.btnAdd.Click += new System.EventHandler(this.btnAdd_Click);
```
**翻訳**: 「追加ボタン（btnAdd）のクリックイベント（Click）に、btnAdd_Clickメソッドを接続する」

### レビューで確認すること

Designer.csを開いて、各ボタンに対する `Click +=` の行があるか確認する。

```csharp
// 問題1のDesigner.csから抜粋:
this.btnAdd.Click += new System.EventHandler(this.btnAdd_Click);        // 追加ボタン → 接続あり
this.btnEdit.Click += new System.EventHandler(this.btnEdit_Click);      // 編集ボタン → 接続あり
this.btnDelete.Click += new System.EventHandler(this.btnDelete_Click);  // 削除ボタン → 接続あり
this.btnSearch.Click += new System.EventHandler(this.btnSearch_Click);  // 検索ボタン → 接続あり
this.btnShowAll.Click += new System.EventHandler(this.btnShowAll_Click);// 全件表示   → 接続あり
this.Load += new System.EventHandler(this.FormMain_Load);               // フォーム読み込み → 接続あり
this.dgvCustomers.SelectionChanged += new System.EventHandler(this.dgvCustomers_SelectionChanged); // 選択変更 → 接続あり
```

**この行がなければ**、ボタンを押しても何も起きない。これは典型的なバグの原因になる。

### Designer.csでもう1つ確認できること: コントロールの種類

Designer.csの先頭付近に、画面部品の宣言がある。

```csharp
private System.Windows.Forms.TextBox txtName;       // txtName はテキストボックス
private System.Windows.Forms.Button btnAdd;          // btnAdd はボタン
private System.Windows.Forms.DataGridView dgvCustomers; // dgvCustomers はデータグリッド
private System.Windows.Forms.Label lblStatus;        // lblStatus はラベル
```

設計仕様書の「画面構成要素」と突き合わせて、必要なコントロールがすべて存在するか確認できる。

## 5. 「処理の流れ」の追い方

設計仕様書の「顧客追加（FR-001）」を例に、実際にコードで処理の流れを追う方法を示す。

### 追跡対象: 仕様FR-001「顧客追加」

仕様書の記述:
- 「追加」ボタン押下で入力欄の値を顧客リストに追加する
- IDは自動採番
- **追加後、入力欄をクリアする**
- **追加後、一覧を更新して新しい顧客が表示されること**

### 追跡の流れ

```
ユーザーが「追加」ボタンをクリック
  |
  v
【Designer.cs で接続を確認】
  this.btnAdd.Click += new System.EventHandler(this.btnAdd_Click);
  → 接続あり。btnAdd_Click メソッドが呼ばれる
  |
  v
【FormMain.cs: btnAdd_Click メソッドに入る】
  |
  +-- 28行目: if (string.IsNullOrWhiteSpace(txtName.Text))
  |           → 氏名が空かチェック
  |           → 空なら「氏名を入力してください」と表示して return（処理中断）
  |           → 仕様FR-005「氏名: 必須」 ... OK、チェックされている
  |
  +-- 35行目: if (!int.TryParse(txtAge.Text, out int age))
  |           → 年齢が整数かチェック
  |           → 整数でなければ「年齢は整数で入力してください」と表示して return
  |           → 仕様FR-005「年齢: 0以上150以下の整数」
  |           → 整数チェックはあるが... 範囲チェック（0〜150）がない!
  |           ========================================
  |           ★ バグ発見: 年齢の範囲チェックが未実装
  |           ========================================
  |
  +-- 46行目: var customer = new Customer(nextId++, ...)
  |           → 顧客オブジェクトを作成（ID自動採番）
  |           → 仕様FR-001「IDは自動採番」 ... OK
  |
  +-- 47行目: customers.Add(customer)
  |           → 顧客リストに追加
  |
  +-- ここで仕様を確認: 「追加後、入力欄をクリアする」
  |           → txtName.Text = "" のような行がない!
  |           ========================================
  |           ★ バグ発見: 追加後の入力欄クリアが未実装
  |           ========================================
  |
  +-- 54行目: RefreshGrid(customers)
              → 一覧を更新
              → 仕様FR-001「追加後、一覧を更新」 ... OK
```

### 追跡のコツ

1. **仕様書を横に開いておく**: コードの各行が仕様のどの項目に対応するかを常に意識する
2. **仕様に書いてあるのに、対応するコードがない**: これがバグ（実装漏れ）
3. **コードに書いてあるが、仕様の記述と一致しない**: これもバグ（実装ミス）
4. **「この処理の次に何があるべきか」を仕様書で確認する**: 上の例では「追加後のクリア」がこのパターン

## 6. VB6経験者向けの対応表

VB6を少し独学した経験があれば、以下の対応表でC#のコードが読みやすくなる。

### 基本構文の対応

| VB6 | C# | 意味 |
|-----|-----|------|
| `Dim x As Integer` | `int x;` | 整数変数の宣言 |
| `Dim s As String` | `string s;` | 文字列変数の宣言 |
| `Dim flag As Boolean` | `bool flag;` | 真偽値変数の宣言 |
| `x = 10` | `x = 10;` | 値の代入（C#は末尾に`;`が必要） |
| `If x > 0 Then ... End If` | `if (x > 0) { ... }` | 条件分岐（C#は`{}`で囲む） |
| `If x > 0 Then ... Else ... End If` | `if (x > 0) { ... } else { ... }` | 条件分岐（else付き） |
| `For i = 0 To 9 ... Next i` | `for (int i = 0; i < 10; i++) { ... }` | 繰り返し |
| `MsgBox "hello"` | `MessageBox.Show("hello")` | メッセージ表示 |
| `MsgBox "確認", vbYesNo` | `MessageBox.Show("確認", ..., MessageBoxButtons.YesNo)` | はい/いいえダイアログ |

### イベント処理の対応

| VB6 | C# | 意味 |
|-----|-----|------|
| `Private Sub cmdAdd_Click()` | `private void btnAdd_Click(object sender, EventArgs e)` | ボタンクリック処理 |
| `Private Sub Form_Load()` | `private void FormMain_Load(object sender, EventArgs e)` | フォーム読み込み処理 |
| コントロール名直結で自動接続 | Designer.csで `Click +=` で接続 | イベントの接続方法 |

### 画面部品の対応

| VB6 | C# (WinForms) | 意味 |
|-----|---------------|------|
| `Me.txtName.Text` | `txtName.Text` | テキストボックスの値（C#では`Me.`不要） |
| `cmdAdd` (CommandButton) | `btnAdd` (Button) | ボタン |
| `txtName` (TextBox) | `txtName` (TextBox) | テキストボックス |
| `lstItems` (ListView) | `dgvCustomers` (DataGridView) | 一覧表示 |
| `lblStatus` (Label) | `lblStatus` (Label) | ラベル |

### その他の対応

| VB6 | C# | 意味 |
|-----|-----|------|
| `' コメント` | `// コメント` | コメント記号が違う |
| `&` (文字列結合) | `+` または `$"...{変数}"` | 文字列の連結方法 |
| `Sub`（戻り値なし） | `void`（戻り値なし） | 戻り値がない処理 |
| `Function ... As Integer` | `int メソッド名(...)` | 戻り値がある処理 |
| `Nothing` | `null` | 「何もない」を表す値 |
| `On Error GoTo` | `try { ... } catch { ... }` | エラー処理 |
| `Set obj = New Class1` | `var obj = new Class1()` | オブジェクトの作成 |
| 行末に記号不要 | 行末に `;` が必要 | 文の終わりの書き方 |
| `And` / `Or` | `&&` / `||` | 論理演算子 |
| `Not` | `!` | 否定 |

### 最大の違い: 波括弧 `{ }`

VB6は `If...Then...End If` のように開始と終了を言葉で書く。C#は `{ }` で囲む。

```
VB6:                              C#:
If age >= 18 Then                 if (age >= 18)
    MsgBox "成人です"              {
End If                                MessageBox.Show("成人です");
                                  }
```

波括弧の対応関係に慣れれば、C#のコードはVB6の知識でかなり読めるようになる。
