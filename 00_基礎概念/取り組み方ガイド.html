BSEコードレビュー訓練 取り組み方ガイド

コードを「書く」のではなく「読んでレビューできる」ことが目標。この教材で段階的に力をつけていく。

対象者

この教材は、ブリッジSE（BSE）としてコードレビューを行う方を対象としている。

  • コーディング経験は少なくても問題ない
  • VB6を少し独学した程度の知識があれば十分
  • オフショア開発チームから納品されるコードをレビューする業務を想定

この教材のゴール: 設計仕様書とソースコードを突き合わせて「仕様と実装の差異（バグ）」を見つけ、開発者に的確な指摘ができるようになること。

このリポジトリの使い方

Step 1: GitHubからcloneする

git clone https://github.com/[リポジトリURL] bse-training
cd bse-training

Git操作に不慣れな場合は、GitHubの「Code」ボタン → 「Download ZIP」でも可。

Step 2: index.htmlをブラウザで開く

リポジトリのルートにある index.html をダブルクリックすると、ポータル画面が開く。
ここから各問題・各ガイドへリンクで移動できる。

Step 3: 推奨順序に沿って進める

下記の「推奨学習順序」に従って、基礎概念の理解 → 実際の問題演習 の順で進める。

推奨学習順序

Step 1: 基礎概念を読む（00_基礎概念/）

最初に以下の3つを順に読む。

順番 ファイル 内容
1-1 CSharp基礎概念.md C#の変数・制御構造・メソッド・クラス・イベント駆動・DB接続・例外処理の基礎。他の言語（VB.NET, VBA等）の理解にも共通する概念が多いため、最初に必ず読む
1-2 コードリーディングガイド.md FormMain.csの読み方を5ステップで解説。頻出コードパターンの日本語翻訳辞書付き。コードが読めない場合はここを先に読む
1-3 VS設定ガイド.md Visual Studio 2022のインストールと基本設定。問題を動かすために必須
1-4 デバッグガイド.md ブレークポイント・ステップ実行・変数確認の方法。バグ発見の最も強力な手段

理由: C#の基礎概念は、この教材のすべての問題の土台になる。Visual Studioとデバッグ操作を先に覚えておくと、問題演習の効率が大幅に上がる。

Step 2: C# Level 1 問題1～4（01_CSharp_Level1/）

この教材で最も問題数が多く、段階的に難易度が上がるように設計されている。

順番 問題 テーマ 難易度

Step 3: SQL Server（07_SQLServer_Level1/）

順番 テーマ
3-1 テーブル設計の問題を見つける
3-2 SQLクエリのバグ（JOIN、GROUP BY等）

理由: C#の問題4（DB接続バグ）でSQL Serverに触れている。その知識が温かいうちにDB側の知識を深める。実務ではDBレビューの比重が大きい。

Step 4: VB.NET（02_VBNET_Level1/）

理由: VB.NETはC#と同じ.NETプラットフォーム上で動作し、構文が似ている。C#の知識があればスムーズに入れる。既存の業務アプリでVB.NETが使われているプロジェクトは多い。

Step 5: VBA（03_VBA_Level1/）

理由: Excel帳票のマクロは実務で頻出する。Excelさえあれば環境構築不要で試せるため、取り組みやすい。

Step 6: Git演習（06_Git/）

理由: コードレビューの実務フローを学ぶ。clone → ブランチ作成 → レビュー → Pull Request の流れを体験する。ここまでの問題で「何をレビューするか」は身についているので、「どうやってレビューするか」のワークフローを覚える。

Step 7: VB6（04_VB6_Level1/）

理由: レガシーアプリの保守案件がある場合に必要。新規開発では使わないが、既存システムの改修レビューで求められることがある。案件に応じて取り組む。

Step 8: C++（05_CPP_Level1/）

理由: CAD案件など、C++が使われるプロジェクトがある場合に取り組む。ポインタやメモリ管理など独特の概念があるため、他の言語とは別枠で学習する。

1問の取り組み方

各問題は以下の10ステップで進める。必ずこの順序で行うこと。

ステップ1: 問題説明.mdを読む

問題の状況設定とタスクを把握する。「バグが何個あるか」のヒントもここに書かれている。

ステップ2: 設計仕様書を熟読する

設計仕様書はレビューの「正解」にあたる。特に以下の部分に注目する。

  • 太字・強調されている箇所: バグが仕込まれているポイントのヒント
  • 画面レイアウト図: ボタンやテキストボックスの配置
  • 機能仕様: 各ボタンの動作、計算ロジック、入力チェック
  • エラー処理: エラーメッセージの文言、表示タイミング

ステップ3: ソースコードをVisual Studioで開く

.sln ファイルをダブルクリックしてVisual Studioで開く。テキストエディタ（VSCode等）でも可。

ステップ4: F5でデバッグ実行し、仕様書の通りに操作する

まずは「普通のユーザー」として操作してみる。仕様書に書かれた操作を1つずつ試す。

ステップ5: 仕様と異なる動作を見つけたらメモする

「仕様書にはこう書いてあるが、実際はこう動く」という差異をメモする。この時点では原因の特定は不要。

ステップ6: ソースコードを読み、バグの箇所を特定する

デバッグ機能（ブレークポイント、ステップ実行）を使って、どの行が原因かを突き止める。デバッグガイドを参考にする。

ステップ7: レビューチェックリストで確認漏れがないかチェック

01_CSharp_Level1/レビューチェックリスト.md を使って、見落としがないか体系的に確認する。

ステップ8: レビュー指摘テンプレートに沿って指摘を書く

見つけたバグを以下のテンプレートで文書化する（詳細は後述の「レビュー指摘の書き方のコツ」を参照）。

【指摘箇所】ファイル名: 行番号付近
【分類】バグ / 仕様不一致 / 改善提案
【重要度】高 / 中 / 低
【内容】設計仕様書では○○と定義されているが、実装では△△となっている。
【修正案】△△を□□に変更してください。

ステップ9: 回答解説で答え合わせ

回答解説/ フォルダの対応ファイルを読み、自分の指摘と比較する。

  • 見つけられなかったバグがあれば、なぜ見落としたかを振り返る
  • 自分の指摘文と回答解説の指摘例文を比較し、書き方を改善する

ステップ10: 修正コードを見て「なぜそう直すのか」を理解する

回答解説には修正後のコードが載っている。「何を」「どう」直したかだけでなく、「なぜその修正が正しいのか」を理解する。これが次の問題でのレビュー力向上につながる。

レビュー指摘の書き方のコツ

基本原則: 事実ベースで書く

悪い例:

ここが間違っています。直してください。

良い例:

【内容】設計仕様書3.2節「検索機能」では「大文字小文字を区別しない」と定義されていますが、
FormMain.cs 45行目の検索処理で String.Contains() を使用しており、
大文字小文字が区別される実装になっています。
【修正案】String.Contains() の第2引数に StringComparison.OrdinalIgnoreCase を
指定してください。

修正案を必ず添える

指摘だけでなく「どう直せばよいか」を具体的に書く。オフショア開発者が指摘を見ただけで修正できるレベルが理想。

重要度の判断基準

重要度 基準 例
高 データ破損、セキュリティ、システム停止の恐れ SQLインジェクション、接続文字列のミス、データ消失
中 機能が正しく動作しない 計算結果が違う、検索できない、追加が反映されない
低 使い勝手の問題、仕様の軽微な差異 確認ダイアログがない、入力欄がクリアされない

オフショア開発者に伝わるように書く

  • ファイル名と行番号を必ず含める
  • 仕様書のどの項目に対する指摘かを明記する
  • 曖昧な表現（「適切に」「正しく」）を避け、具体的な値や条件を書く
  • 必要に応じてコード例を添える

困ったときは

バグが見つからない場合

  1. デバッグで値を確認する: ブレークポイントを設置してF5実行。変数の値が期待通りか確認する（デバッグガイド参照）
  2. 仕様書の太字・強調部分を再確認する: バグのヒントが隠れていることが多い
  3. レビューチェックリストを上から順に確認する: 体系的に確認することで見落としを防ぐ

ビルドエラーが出る場合

  1. Visual Studioの「エラー一覧」ウィンドウで内容を確認する
  2. .NET 8 SDK がインストールされているか確認する
  3. VS設定ガイドの「トラブルシューティング」を参照する

コードの読み方がわからない場合

  1. 00_基礎概念/コードリーディングガイド.md で「コードの読み方5ステップ」と「翻訳辞書」を確認する
  2. 00_基礎概念/CSharp基礎概念.md に戻って該当する文法を確認する
  3. Visual Studioで変数やメソッドにカーソルを合わせると型情報が表示される
  4. F12（定義へ移動）でメソッドの中身を確認する

レビュー指摘の書き方に迷う場合

  1. 回答解説の「レビュー指摘例文」を参考にする
  2. 上記テンプレートに当てはめて書く
  3. 「仕様書では○○、実装は△△、修正案は□□」の3点を必ず含める
