# デバッグガイド（BSE向け）

> プログラムを1行ずつ実行して変数の値を確認する。仕様通りに動いているか確かめる最も確実な手段。

---

## 1. デバッグとは何か

デバッグとは、プログラムを1行ずつ実行しながら、変数の値や処理の流れを確認する作業のこと。

### なぜBSEにデバッグが必要か

- コードを読むだけでは、実際にどんな値が変数に入っているかわからない
- デバッグを使えば「仕様書の期待値」と「実際の値」を直接比較できる
- バグの原因を「ここがおかしい気がする」ではなく「この行で値が期待と異なる」と特定できる

**一言で言えば**: デバッグは「仕様通りに動いているか確認する手段」。コードを書く能力がなくても使える。

### ブレークポイントとは

プログラムの実行を「ここで止めてほしい」と指定するマーク。ブレークポイントを置いた行でプログラムが一時停止し、その時点の変数の値を確認できる。

### 設置方法

**方法1**: 止めたい行にカーソルを置いて **F9** キーを押す（もう一度押すと解除）

**方法2**: コードエディタの行番号の左側（余白部分）をクリックする → 赤い丸が表示される

```
設置後のイメージ:

    43      private void btnAdd_Click(object sender, EventArgs e)
    44      {
  ● 45          string name = txtName.Text;     ← ここで止まる（●が赤い丸）
    46          string phone = txtPhone.Text;
    47          AddCustomer(name, phone);
```

### 条件付きブレークポイント

特定の条件を満たすときだけ止めたい場合に使う。

1. ブレークポイント（赤い丸）を右クリックする
2. 「条件」を選択する
3. 条件式を入力する（例: `i == 5`、`name == "田中"`）

**使いどころ**: ループの中で100回繰り返す処理があるが、5回目だけ確認したい場合など。

### ブレークポイントの一覧管理

メニューの「デバッグ」→「ウィンドウ」→「ブレークポイント」で一覧を表示できる。
設置した全ブレークポイントの有効/無効を切り替えたり、一括削除したりできる。

### ブレークポイントを置くべき場所のコツ

| 場所 | 理由 |
|------|------|
| ボタンクリックのイベントハンドラの先頭 | ボタンを押したとき、このメソッドが呼ばれるか確認できる |
| if文の直前 | 条件式の値が期待通りか確認できる |
| 計算処理の前後 | 計算の入力値と結果が正しいか確認できる |
| DB操作の前後 | SQLのパラメータ値と取得結果を確認できる |
| メソッド呼び出しの前後 | メソッドに渡す引数と返ってくる値を確認できる |

## 3. ステップ実行

ブレークポイントで止まった後、1行ずつ実行を進める操作。

### 3つのステップ操作

| 操作 | キー | 動作 |
|------|------|------|
| ステップオーバー | **F10** | 次の行に進む。メソッド呼び出しがあっても中には入らない |
| ステップイン | **F11** | 次の行に進む。メソッド呼び出しがあれば中に入る |
| ステップアウト | **Shift+F11** | 今いるメソッドの残りを一気に実行して、呼び出し元に戻る |

### 実行続行

| 操作 | キー | 動作 |
|------|------|------|
| 続行 | **F5** | 次のブレークポイントまで一気に実行する |
| 停止 | **Shift+F5** | デバッグを終了する |

```
コード:
    45  string name = txtName.Text;
    46  string phone = txtPhone.Text;
    47  AddCustomer(name, phone);        ← 今ここで止まっている
    48  RefreshGrid();

    AddCustomer メソッドの中身:
        60  var customer = new Customer(name, phone);
        61  customers.Add(customer);
        62  return;


【F10（ステップオーバー）を押した場合】

    47  AddCustomer(name, phone);        ← AddCustomerの中身は実行されるが
    48  RefreshGrid();                   ← 止まるのはここ（次の行）

    → メソッドの中身には入らず、結果だけ得て次へ進む


【F11（ステップイン）を押した場合】

    47  AddCustomer(name, phone);        ← ここから
        60  var customer = new ...        ← AddCustomerの中のここに入る

    → メソッドの中身に入って、1行目で止まる
```

**使い分けの目安**:
- まず **F10** で全体の流れを確認する
- 怪しいメソッドが見つかったら、もう一度実行して今度は **F11** で中に入る
- 興味のないメソッドの中に入ってしまったら **Shift+F11** で抜ける

## 4. 変数の値を確認する方法

### 方法1: マウスホバー（最も手軽）

デバッグ中に停止した状態で、変数にマウスカーソルを合わせるとツールチップで値が表示される。

```
    45  string name = txtName.Text;
                     ↑ カーソルを合わせると "田中太郎" と表示される
```

オブジェクトの場合は展開できる。`customer` にホバーすると、`Id`, `Name`, `Phone` 等の各プロパティの値を確認できる。

### 方法2: ローカルウィンドウ

メニューの「デバッグ」→「ウィンドウ」→「ローカル」で開く。
現在のスコープにあるすべての変数とその値が一覧表示される。

```
ローカルウィンドウ:
  名前          値                型
  name          "田中太郎"        string
  phone         "090-1234-5678"  string
  customers     Count = 3        List<Customer>
```

**利点**: いちいちマウスを合わせなくても、すべての変数が見える。

### 方法3: ウォッチウィンドウ

メニューの「デバッグ」→「ウィンドウ」→「ウォッチ」→「ウォッチ 1」で開く。
監視したい変数や式を登録しておくと、ステップ実行のたびに値が更新される。

```
ウォッチウィンドウ:
  名前                  値        型
  customers.Count       3         int
  txtName.Text          "田中"    string
  price * taxRate       198.0     decimal
```

**利点**: 特定の変数だけを継続的に監視できる。式（計算結果）も登録できる。

### 方法4: イミディエイトウィンドウ

メニューの「デバッグ」→「ウィンドウ」→「イミディエイト」で開く。
C#の式を直接入力して、その場で評価（実行）できる。

```
イミディエイトウィンドウ:
> customers.Count
3
> name.ToUpper()
"田中太郎"
> price * 1.10m
2178.55
> customers[0].Name
"鈴木一郎"
```

**利点**: 変数の値だけでなく、メソッドの呼び出し結果や計算結果をその場で確認できる。「この式の結果はどうなるか」を試すのに最適。

### 方法5: DataTips

マウスホバーで表示されるツールチップの右側にあるピンアイコンをクリックすると、DataTipが固定表示される。コードの横に値が常に表示された状態で、ステップ実行するたびにリアルタイムで更新される。

## 5. デバッグの実践例

### 例1: 「追加ボタンを押してもリストに反映されない」

この教材の問題1に近いパターン。

**手順**:

1. `btnAdd_Click` メソッドの先頭にブレークポイントを設置する（F9）

```csharp
  ● private void btnAdd_Click(object sender, EventArgs e)
    {
        string name = txtName.Text;
```

2. F5でデバッグ実行する
3. アプリが起動したら、名前と電話番号を入力して「追加」ボタンを押す
4. ブレークポイントで止まるか確認する
   - **止まらない場合**: イベントハンドラが接続されていない（Designer.csを確認）
   - **止まった場合**: 次のステップへ
5. F10でステップ実行しながら、`customers.Count` の値を確認する
   - `customers.Add(customer)` の前後で Count が増えているか
6. `RefreshGrid()` メソッドの中に入り（F11）、`dgvCustomers.DataSource` に正しいデータが設定されているか確認する

### 例2: 「計算結果が合わない」

この教材の問題3に近いパターン。

**手順**:

1. 計算メソッドにブレークポイントを設置する

```csharp
  ● private decimal CalculateTotal(decimal price, int quantity, decimal taxRate)
    {
        decimal subtotal = price * quantity;
        decimal tax = subtotal * taxRate;
        decimal total = subtotal + tax;
        return total;
    }
```

2. F5でデバッグ実行し、計算が実行される操作を行う
3. F10で1行ずつ進めながら、各変数の値を確認する

```
ステップ実行での値の変化:

  行    変数           期待値     実際の値     判定
  ---   subtotal       1000       1000         OK
  ---   tax            100        100          OK
  ---   total          1100       1100         OK（ここまではOK）
```

4. 期待値と実際の値が異なる行がバグの原因
5. 式の各部分をイミディエイトウィンドウで確認する

```
イミディエイトウィンドウ:
> price
1000
> quantity
1
> taxRate
0.10
> price * quantity
1000
> subtotal * taxRate
100
```

### 例3: 「検索結果がおかしい」

この教材の問題2に近いパターン。

**手順**:

1. `btnSearch_Click` にブレークポイントを設置する
2. F5で実行し、検索キーワードを入力して検索ボタンを押す
3. ブレークポイントで止まったら、keyword変数の値を確認する

```
> keyword
"tanaka"
```

4. 検索処理（FindAll等）の結果をウォッチウィンドウで確認する

```
ウォッチ:
  customers.FindAll(c => c.Name.Contains(keyword))    Count = 0
```

5. Count が 0 なら「見つからない」ということ。データ内の名前を確認する

```
> customers[0].Name
"Tanaka"
```

6. "tanaka" と "Tanaka" → 大文字小文字の違いが原因とわかる
7. `Contains` の引数に `StringComparison.OrdinalIgnoreCase` が必要

## 6. よくあるデバッグのパターン

| 症状 | 確認すべきこと | 確認方法 |
|------|-------------|---------|
| ボタンが反応しない | イベントハンドラが接続されているか | Designer.csで `+=` の行を検索。ブレークポイントが止まるか確認 |
| 値が入力と違う | 型変換が正しいか | 変換前後の値をウォッチで確認。`int.TryParse` の戻り値を確認 |
| 計算が合わない | 演算子と型が正しいか | イミディエイトウィンドウで式の各部分を計算。整数除算（5/2=2）に注意 |
| 一覧が更新されない | DataSourceの再設定があるか | RefreshGrid系メソッドにF11で入り、DataSource設定を確認 |
| エラーで落ちる | try-catchがあるか、null参照がないか | 例外が発生する行にブレークポイント。変数がnullでないか確認 |
| 検索でヒットしない | 比較方法が正しいか | 検索キーワードとデータの値をウォッチで並べて比較 |
| 削除しても消えない | リストから実際に削除されているか | Remove/RemoveAtの前後でCount確認。画面更新処理の有無確認 |
| 保存されない | SQLのパラメータが正しいか | SqlCommandのParameters内の値をウォッチで確認 |

## 7. デバッグなしでのレビュー（VB6やコードだけのレビュー用）

Visual Studioが使えない環境や、VB6などデバッグ環境の構築が難しい場合、コードを読むだけでレビューする必要がある。

### コードを読みながら脳内で値をトレースする方法

1. メソッドの入口で、引数にどんな値が入るか想定する
2. 1行ずつ読み、変数の値がどう変わるか頭の中で追う
3. if文では「この条件は真か偽か」を判断して、どちらの分岐に進むか追う
4. 仕様書の期待値と、追跡した値が異なる箇所を探す

### 紙とペンで変数の値を追跡するテクニック

複雑な処理や、ループがある場合は紙に書くのが確実。

```
追跡例: 合計金額の計算

変数追跡表:
  行   |  i  |  price[i]  |  total
  -----|-----|------------|--------
  初期 |  -  |     -      |    0
  1周目|  0  |   1000     |  1000
  2周目|  1  |   2000     |  3000
  3周目|  2  |    500     |  3500
  ←ここで期待値は3500。仕様書と一致するか？
```

### 条件分岐表を書いて網羅的に確認する方法

if-elseやswitch文が多い処理では、すべての条件パターンを洗い出して表にする。

```
条件分岐表: 割引率の計算

  会員ランク | 購入金額    | 期待する割引率 | コードの結果 | 一致？
  -----------|------------|---------------|-------------|-------
  ゴールド   | 10000以上  | 20%           | 20%         | OK
  ゴールド   | 10000未満  | 10%           | 10%         | OK
  シルバー   | 10000以上  | 10%           | 10%         | OK
  シルバー   | 10000未満  | 5%            | 0%          | NG ← バグ発見
  一般       | 金額問わず | 0%            | 0%          | OK
```

この方法を使えば、デバッグ実行なしでも論理的にバグを見つけられる。特に境界値（ちょうど10000、0、マイナス値等）を漏れなく確認することが重要。

### 基本の流れ

```
1. 仕様書を読んで「期待する動作」を把握する
            ↓
2. 怪しい箇所にブレークポイントを置く（F9）
            ↓
3. デバッグ実行する（F5）
            ↓
4. アプリを操作してブレークポイントで止める
            ↓
5. 変数の値を確認する（ホバー/ローカル/ウォッチ）
            ↓
6. ステップ実行で1行ずつ進める（F10/F11）
            ↓
7. 「期待値」と「実際の値」が異なる行を見つける
            ↓
8. バグの原因を特定し、レビュー指摘を書く
```

### 覚えるべきキー（最低限これだけ）

| キー | 操作 | 一言 |
|------|------|------|
| F9 | ブレークポイント | 止めたい行に赤丸を置く |
| F5 | デバッグ実行/続行 | 走らせる。赤丸で止まる |
| F10 | ステップオーバー | 次の行へ。メソッドには入らない |
| F11 | ステップイン | 次の行へ。メソッドの中に入る |
| Shift+F5 | デバッグ停止 | デバッグを終了する |

この5つのキーを使いこなせれば、この教材の全問題に対応できる。
