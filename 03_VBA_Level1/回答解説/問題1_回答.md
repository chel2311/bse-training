### バグ1: 売上金額をE列の値そのまま使っている（計算ロジック誤り）

**該当仕様**: FR-002 売上金額の再計算
> **売上金額は「数量(C列) x 単価(D列)」で計算する**
> **E列の値（手入力値）をそのまま使用してはならない**

**該当コード**: `SalesReport.bas` 転記処理部分

```vba
' 現在の実装（バグ）
wsReport.Cells(outputRow, 5).Value = wsData.Cells(i, 5).Value       ' 売上金額

' --- 合計用に加算 ---
totalQty = totalQty + wsData.Cells(i, 3).Value
totalAmount = totalAmount + wsData.Cells(i, 5).Value
' → E列の手入力値をそのまま転記・合計している
' → サンプルデータの「コピー用紙A4(1月)」は数量50 x 単価450 = 22,500が正しいのに、E列は25,500
' → サンプルデータの「ホチキス」は数量20 x 単価350 = 7,000が正しいのに、E列は7,700
```

**修正コード**:
```vba
' 数量 x 単価 で再計算して転記
Dim calcAmount As Currency
calcAmount = wsData.Cells(i, 3).Value * wsData.Cells(i, 4).Value

wsReport.Cells(outputRow, 5).Value = calcAmount                     ' 売上金額（再計算）

' --- 合計用に加算 ---
totalQty = totalQty + wsData.Cells(i, 3).Value
totalAmount = totalAmount + calcAmount
```

### バグ2: 合計行が太字になっていない（書式設定漏れ）

**該当仕様**: FR-005 合計行の書式設定
> **合計行のフォントを太字（Bold）にする**
> 対象範囲: 合計行のA列～E列

**該当コード**: `SalesReport.bas` 合計行追加後の処理

```vba
' 現在の実装（バグ）
' --- FR-004: 合計行の追加 ---
wsReport.Cells(outputRow, 2).Value = "合計"
wsReport.Cells(outputRow, 3).Value = totalQty
wsReport.Cells(outputRow, 5).Value = totalAmount

' → ここでFR-005の太字設定処理が完全に欠落している
' → 直後に完了メッセージに進んでしまう

' --- 完了メッセージ ---
Application.ScreenUpdating = True
```

**修正コード**:
```vba
' --- FR-004: 合計行の追加 ---
wsReport.Cells(outputRow, 2).Value = "合計"
wsReport.Cells(outputRow, 3).Value = totalQty
wsReport.Cells(outputRow, 5).Value = totalAmount

' --- FR-005: 合計行を太字にする ---
wsReport.Range(wsReport.Cells(outputRow, 1), wsReport.Cells(outputRow, 5)).Font.Bold = True

' --- 完了メッセージ ---
Application.ScreenUpdating = True
```

**レビュー指摘例**:
```
【指摘箇所】SalesReport.bas: FR-004合計行追加処理の直後
【分類】仕様不一致
【重要度】中
【内容】設計仕様書 FR-005 では「合計行のフォントを太字（Bold）にする」と定義されているが、
        実装では合計行のデータ設定後に太字設定の処理が行われていない。
        帳票としての視認性に影響があり、印刷時に合計行が目立たない。
【修正案】合計行のデータ設定後に以下のコードを追加してください。
        wsReport.Range(wsReport.Cells(outputRow, 1), wsReport.Cells(outputRow, 5)).Font.Bold = True
```

### バグ3: 既存データのクリアが不完全（固定範囲のみクリア）

**該当仕様**: FR-007 既存データのクリア
> **処理開始前に「月次帳票」シートの既存データをすべてクリアする**
> 前回実行分のデータが残らないよう、**UsedRange全体をクリア**する

**該当コード**: `SalesReport.bas` クリア処理

```vba
' 現在の実装（バグ）
' --- FR-007: 既存データのクリア ---
wsReport.Range("A1:E50").Clear
' → 固定範囲（50行まで）しかクリアしていない
' → 前回のデータが51行以上あった場合、51行目以降のデータが残る
' → 例: 前回100件のデータを生成→今回10件のデータを生成→11行目以降に前回の90件が残る
```

**修正コード**:
```vba
' --- FR-007: 既存データのクリア ---
wsReport.UsedRange.Clear
' → 使用されている範囲全体をクリアする
' → データ量に関わらず確実に前回データを削除できる
```

**レビュー指摘例**:
```
【指摘箇所】SalesReport.bas: wsReport.Range("A1:E50").Clear の行
【分類】仕様不一致
【重要度】高
【内容】設計仕様書 FR-007 では「UsedRange全体をクリアする」と定義されているが、
        実装では Range("A1:E50") という固定範囲のみをクリアしている。
        前回の実行で50行を超えるデータが生成された場合、51行目以降のデータが
        クリアされずに残り、新しい帳票の下に前回のデータが混在する。
        これは帳票の信頼性に関わる重大な不具合である。
【修正案】wsReport.Range("A1:E50").Clear を wsReport.UsedRange.Clear に変更してください。
        UsedRangeはシート上で使用されている範囲全体を動的に参照するため、
        データ量に関わらず確実にクリアできます。
```

## バグの見つけ方のコツ

1. **仕様書のFR番号を1つずつコードと照合する**
   - FR-001からFR-008まで順にチェック。今回のバグは FR-002, FR-005, FR-007
   - 仕様書に太字で強調された部分は特に注意

2. **「計算」が絡む部分は特に注意する**
   - バグ1: 仕様では「再計算」と明記しているのに、入力値をそのまま使っている
   - 「どの値を使って計算しているか」を追跡することが重要

3. **実際にExcelで動かして確認する**
   - バグ1: サンプルデータの「コピー用紙A4」（E列: 25,500 vs 再計算: 22,500）や「ホチキス」（E列: 7,700 vs 再計算: 7,000）を比較
   - バグ2: 帳票の合計行が太字になっているか目視確認
   - バグ3: 大量データで1回実行→少量データで再実行して、前回データが残らないか確認

4. **「前回の状態」を意識する**
   - バグ3のようなクリア漏れは、1回目の実行では問題にならない
   - 2回以上実行して初めて発覚するバグも多い
   - BSEレビューでは「連続実行」のテストケースを必ず確認する
